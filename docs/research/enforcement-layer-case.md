# The case for the enforcement layer

Date: 2026-07-18. Question: does forge need its non-prose enforcement tiers at all? The tiers decision ([#37](https://github.com/alp82/forge/issues/37)) assumed the enforcement layer is forge's rarest asset; this survey stress-tests that assumption against markdown-only prior art - chiefly mattpocock/skills, which ships no hooks and works well in the driving dev's own use - before the adapter contract ([#40](https://github.com/alp82/forge/issues/40)) bakes the tiers in. Wayfinder ticket #49; also feeds the opencode install path decision (#50).

Primary sources only: this repo's `hooks/` source and `docs/research/pipeline-audit.md` measurements, the mattpocock/skills issue tracker, the obra/superpowers release notes, official Claude Code docs, and this machine's installed state - all read or fetched 2026-07-18 unless dated otherwise. Sibling docs `landscape-survey.md`, `pipeline-audit.md`, and `distribution-prior-art.md` are cross-referenced, not re-derived. House style: hyphens only.

## Executive summary

**Keep the tiers.** The strongest single piece of evidence is the counter-example's own issue tracker: mattpocock/skills - the markdown-only collection cited as proof that enforcement is unnecessary - carries four open issues documenting exactly the failure class forge's stop-gates exist to catch, including "/implement silently skips its /tdd and /code-review steps" ([#479](https://github.com/mattpocock/skills/issues/479)) and "wayfinder: Chart the map often skips /domain-modeling" ([#556](https://github.com/mattpocock/skills/issues/556)). The resolution of the apparent paradox ("Matt's skills work great with no plugin") is a category distinction: a **reference skill** - invoked on demand, teaching how to do X well - needs no enforcement, because nothing is owed when it is not invoked; a **process guarantee** - "the review happens every time, the tests pass before stop" - is precisely the class where prose demonstrably drifts. Pocock's skills work great *as references* without hooks and drift *as process* without hooks, and his tracker records both halves. forge's product claim is the process half. The measured base rate is material: the pipeline audit counted 73 "Stop hook feedback" block events in this repo's transcripts (117 across seven repos, fail message in 20+ sessions) - each one a turn that would otherwise have ended with failing tests, a broken build, or unreviewed code. Prior art agrees from the other direction: superpowers spent four major versions escalating prose countermeasures against "observed agent behavior where they rationalize around skill usage despite clear instructions," and Anthropic's own hooks doc states the rationale in one sentence: hooks ensure "certain actions always happen rather than relying on the LLM to choose to run them." What the evidence does *not* support is growing the layer: forge's six hooks are already the survivors of a cull from 14 registrations, prose is tier-invariant by decision, and the honest recommendation to #40 is to keep the hook surface exactly this small. One measurement caveat reported below: the local git-block log turns out to be test-suite contaminated and cannot serve as live frequency evidence.

## 1. What forge's hooks actually buy

Six hooks ship today (`hooks/hooks.json`: 4 event types, 6 scripts). Per hook: the failure class it prevents, the measured frequency where one exists, and the honest cost of losing it.

| Hook | Event | Failure class prevented | Measured frequency | Cost of prose-only |
|---|---|---|---|---|
| `session-start.sh` | SessionStart (startup/resume/clear/compact) | Session begins with no forge entry rule loaded; stale/dangling skill installs go unnoticed | Fires every session by construction | Entry discipline depends on the user's CLAUDE.md; stale-install detection is simply gone (no prose can run a version check) |
| `block-git-writes.sh` | PreToolUse(Bash) | Destructive git (`reset --hard`, `push --force`, `checkout --`) executed by the agent | Log contaminated - see below | One bad command is unrecoverable work loss; prose can forbid it but cannot intercept it |
| `mark-code-change.py` | PostToolUse(Edit\|Write) | None itself - arms the per-session change markers the three Stop gates read | n/a | Without tracking, gates cannot exist (chat-only turns must skip) |
| `verify-tests.py` | Stop | Turn ends with a failing suite | Together: 73 block events in plugin-repo transcripts, 117 across seven repos; "Tests are failing. Fix them before completing." in 20+ plugin-repo sessions (pipeline-audit § 3) | Failing code ships as "done"; the user discovers it next session |
| `verify-build.py` | Stop | Turn ends with a broken build/typecheck | (counted with above; median cost 23 ms) | Same class, compile-error flavor |
| `review-owed.py` | Stop | Code changed, no review wave ran - the session isn't done and doesn't know it | Newest hook (2.0.x), no transcript corpus yet | The exact drift mattpocock/skills [#479](https://github.com/mattpocock/skills/issues/479) documents (§ 2) |

Three load-bearing facts from the audit (`docs/research/pipeline-audit.md`, measured over 42 plugin-repo sessions, 2026-07):

- **The gates block for real and cheaply**: verify-tests median 2.3 s, p90 5.8 s; verify-build median 23 ms; 825 Stop firings in the plugin repo. The audit's verdict: "Deterministic, cheap... and they block for real," naming this "the Instruction-to-hook principle earning its keep."
- **The six hooks are the survivors of a diet**: alp-river 1.4 ran 8 hook events with 14 registrations; the audit found hook overhead "small in absolute terms" and the 2.0 rebuild kept only these.
- **Prose drifts locally too**: the audit records that a 516-line WORKFLOW.md "had to be cut toward ~250 because stages and users were ignoring parts of it," including phantom machinery no stage implemented - this repo's own history of instructions rotting without a mechanism.

**Measurement honesty - the git-block log.** `~/.claude/debug/git-block.log` holds 2,521 BLOCKED lines, which would be dramatic frequency evidence - but 2,269 of them land on 2026-07-17 across only 31 distinct minutes total, and the blocked commands (`if false; then :; elif git reset --hard; then :; fi`, `coproc git reset --hard`) are recognizably the fixtures of `hooks/tests/test_block_git_writes.py`, which invokes the real script (writing to the real `$HOME` log). The log measures the test suite, not live sessions, and this survey does not cite it as live frequency. Genuine live blocks exist in the log but cannot be cleanly separated; the guard's value case rests on severity (unrecoverable loss) times nonzero incidence, not on a measured rate.

## 2. The Pocock comparison: reference skills vs process guarantees

mattpocock/skills is markdown-only by construction - no `hooks/` directory, payload "markdown-only - no hooks, so nothing is lost on the channel that cannot carry hooks" (`distribution-prior-art.md` § 2) - and massively adopted (175,897 stars, pushed 2026-07-17, per [api.github.com/repos/mattpocock/skills](https://api.github.com/repos/mattpocock/skills)). The driving dev runs 23 of these skills via skills.sh with no plugin installed (`~/.agents/skills/`, verified locally 2026-07-18) and reports they work well. Both facts are real; neither is evidence against enforcement, for two reasons.

**What the skills maintain without enforcement.** Read locally (`~/.agents/skills/tdd/SKILL.md`, `code-review`, `wayfinder`, `grilling`): the discipline is genuine but *informational* - crisp rules stated in prose ("No test is written at an unconfirmed seam," "Red before green"), reference tables, anti-pattern lists. A reference skill's failure mode when ignored is mild: the agent does X less well. Nothing is *owed* across the session; there is no standing invariant a Stop could check. For this class, prose is the correct and complete mechanism - which is exactly why the collection feels like it "works really good" with no plugin: its dominant use is reference.

**Where the same collection demonstrably drifts.** The moment Pocock's skills compose into *processes* - one skill promising to invoke another - the tracker records the drift, in the present, on current models:

- [#479](https://github.com/mattpocock/skills/issues/479) (open, 2026-07-09): "/implement silently skips its /tdd and /code-review steps."
- [#367](https://github.com/mattpocock/skills/issues/367) (open, 2026-06-23): "/implement skips /tdd invocation and writes tests directly."
- [#556](https://github.com/mattpocock/skills/issues/556) (open, 2026-07-13): "wayfinder: Chart the map often skips /domain-modeling."
- [#450](https://github.com/mattpocock/skills/issues/450) (open, 2026-07-04): "wayfinder: work-through-map skips grilling - 'if in doubt' is too vague for task tickets."

"Silently skips" is verbatim the failure `review-owed.py` exists to convert into a loud block. These are open issues with no mechanical fix available - a markdown-only distribution has no hook to reach for, so the only lever is ever-more-forceful prose (see § 3 for how far that escalation goes).

**The category distinction, stated once.** Enforcement is not a property a skill collection needs or doesn't; it is a property of the *claim* being made. "Here is how to do TDD well" needs no gate. "A review runs over every change before the session ends" is a guarantee, and a guarantee backed only by prose is - per the tiers decision's own leitwort - "the review happens every time, not just when the model remembers," demoted to "when the model remembers." forge's differentiator is the guarantee class (`landscape-survey.md` § 2: "enforcement is alp-river's rarest asset: almost the entire landscape relies on prose discipline"). Dropping the tiers would not make forge more like Pocock's skills; it would make forge's central claim false in the same way [#479](https://github.com/mattpocock/skills/issues/479) is false today.

## 3. Prior art: what happens to prose-only discipline at scale

**superpowers - the escalation record.** obra/superpowers (v6.1.1, 2026-07-02) is the adoption leader among agent methodologies and the best longitudinal record of prose-only enforcement failing, because its release notes narrate each round. v3.2.2 (2025-10-21) added a "Common Rationalizations section with 8 specific evasion patterns" ("'This is just a simple question' → WRONG") plus "absolute language": "If even 1% chance a skill applies, you MUST read it... You do not have a choice. You cannot rationalize your way out." v4.0.0 (2025-12-17) added more patterns ("'I know what that means' → Knowing the concept ≠ using the skill"). v6.0.0 moved to "hard gates, a mandatory checklist, and a graphviz process flow to enforce compliance" and an `EnterPlanMode` intercept - changes the notes justify as addressing "observed agent behavior where they rationalize around skill usage despite clear instructions" ([RELEASE-NOTES.md](https://github.com/obra/superpowers/blob/main/RELEASE-NOTES.md)). Two readings, both load-bearing: (a) three-plus rounds of escalating prose did not close the gap, on the same current-generation models the "prose is enough now" argument appeals to; (b) even superpowers' mechanical floor is thin - a session-start injection hook plus prose gates, roughly forge's *guarded* tier - and it still ships a `hooks/` directory and per-harness plugins rather than bare markdown ([github.com/obra/superpowers](https://github.com/obra/superpowers)). The methodology with the most adoption chose mechanism where it could get it.

**The platform's own rationale.** Claude Code's hooks guide states the division of labor in one sentence: hooks "provide deterministic control over Claude Code's behavior, ensuring certain actions always happen rather than relying on the LLM to choose to run them" ([code.claude.com/docs/en/hooks-guide](https://code.claude.com/docs/en/hooks-guide), fetched 2026-07-18). Every surveyed harness that grew a plugin channel grew hook support with it - Claude Code, codex, gemini, Cursor all carry hooks in their plugin payloads (`distribution-prior-art.md` § 5) - which is the ecosystem pricing the same judgment in: skills alone were not enough, or the channels would be markdown-only.

**The counter-evidence, weighed honestly.** Three things argue for dropping the tiers, and none survives contact with the record. *"Matt's skills work with no plugin"* - true for the reference class, refuted by his own tracker for the process class (§ 2). *"Models are getting better at following instructions"* - superpowers' escalation happened 2025-10 through 2026-07, i.e. across the current model generation, and the four Pocock skip-issues are dated 2026-06/07. The trend is real but has not yet retired the failure class, and forge's tiers already price the trend correctly: if prose someday suffices, prose-only becomes the universal tier at zero redesign cost, because prose is tier-invariant by decision (#37: briefs state the full discipline identically at every tier). *"The tiers cost three adapter builds and a contract"* - the cost is real but bounded: the layer is six small scripts (the survivors of the 14-registration diet), the capability checklist is four items, and the alternative - deleting the layer - forfeits the one asset the landscape survey found nobody else has.

## 4. What this feeds

- **The adapter contract (#40)**: keep the tiers and the four-capability checklist as decided in #37; the evidence adds one instruction - treat the current six-hook surface as a ceiling, not a floor. Every hook must name its failure class and (where possible) its measured catch rate, the way § 1's table does; a hook that cannot is a candidate for deletion, not porting.
- **The opencode install path (#50)**: unblocked with the tiers intact - opencode's guarded tier (tool guard + idle nudge) is worth carrying, so the JS plugin needs a delivery path; the question does not evaporate.
- **The distribution hierarchy (#39)**: stands as decided - native channels stay canonical *because* they carry hooks; the skills.sh route stays the loud prose-only tier. No shift toward installer-first.
