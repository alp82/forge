# opencode extension-surface survey

Date: 2026-07-17. Question: what extension surface does opencode (SST's open-source AI coding agent) offer a forge adapter - instruction files, subagents, hooks/plugins, model selection, distribution - and what can it not do relative to Claude Code's plugin surface? Wayfinder ticket #34.

Primary sources only: the official docs at [opencode.ai/docs](https://opencode.ai/docs/) (docs site self-reports "Last updated: Jul 17, 2026"; all pages fetched 2026-07-17) and the [sst/opencode](https://github.com/sst/opencode) repo source on the `dev` branch (fetched 2026-07-17; latest release [v1.18.3, 2026-07-16](https://github.com/sst/opencode/releases)). Load-bearing claims about hooks and schemas are verified against the TypeScript source, not just prose docs. Secondary lists (awesome-opencode, opencode.cafe) appear only as discovery indexes, labeled as such. House style: hyphens only. The findings feed three decisions: (1) whether subagent spawning is a required adapter capability opencode can supply, (2) what enforcement tier opencode supports (any Stop-hook equivalent?), (3) how to distribute forge there.

## Executive summary

Four findings drive the three decisions. First, **subagent spawning is fully supported, with per-task model choice**: markdown agent definitions with `mode: subagent` carry a `model` frontmatter override, are invoked automatically by description, manually by @-mention, or programmatically via the task tool ([docs/agents](https://opencode.ai/docs/agents/)), and the server API can spawn true child sessions (`POST /session` with `parentID`, `GET /session/:id/children`) with a per-prompt model override ([docs/server](https://opencode.ai/docs/server/)). Forge's spawn requirement is met. Second, **there is no Stop-hook equivalent - blocking the end of a session is IMPOSSIBLE**: the plugin `Hooks` interface ([packages/plugin/src/index.ts](https://github.com/sst/opencode/blob/dev/packages/plugin/src/index.ts)) contains no session-lifecycle hook that can veto or gate anything; `session.idle` is a fire-and-forget notification. The enforcement ceiling is tool-level: `tool.execute.before` can throw to block a tool call, `permission.ask` can force-deny, and a plugin can reactively re-prompt an idle session via the SDK client - a nudge after the fact, not a gate. The old `experimental.hook` config (`session_completed`, `file_edited`) is gone from the codebase entirely. Third, **context injection is strong**: plugins can mutate every user message (`chat.message` receives mutable `parts`), transform the system prompt (`experimental.chat.system.transform`), and the rules system loads `AGENTS.md` plus arbitrary `instructions` globs ([docs/rules](https://opencode.ai/docs/rules/)). Fourth, **there is no marketplace**: distribution is npm for JS plugins plus plain file conventions for agents/commands/skills - and opencode natively reads `.claude/skills/` and `~/.claude/skills/` SKILL.md directories ([docs/skills](https://opencode.ai/docs/skills/)), so forge's skill files are loadable as-is; only the hook enforcement layer has no home.

## 1. Instruction surface

### 1.1 Rules files (AGENTS.md)

Per [docs/rules](https://opencode.ai/docs/rules/) (fetched 2026-07-17): project rules live in `AGENTS.md` at the project root (applies to that directory tree); global personal rules in `~/.config/opencode/AGENTS.md`. Search order on init: local files found by traversing upward from cwd (`AGENTS.md`, then `CLAUDE.md`), then the global file, then the Claude Code compatibility fallback `~/.claude/CLAUDE.md` - "the first matching file wins in each category", so a project `AGENTS.md` shadows a project `CLAUDE.md`. Claude Code compatibility (project `CLAUDE.md`, global `~/.claude/CLAUDE.md`, `~/.claude/skills/`) can be disabled via `OPENCODE_DISABLE_CLAUDE_CODE=1`.

Beyond the fixed files, `opencode.json` takes an `instructions` array of relative paths, glob patterns (`.cursor/rules/*.md`), or remote URLs (fetched with a 5-second timeout); "all instruction files are combined with your `AGENTS.md` files" ([docs/rules](https://opencode.ai/docs/rules/)). This is a clean channel for an adapter to inject forge doctrine files without owning `AGENTS.md`. opencode does not auto-follow file references inside `AGENTS.md` - the docs recommend the `instructions` array or teaching the model to load files on demand.

### 1.2 Custom commands

Per [docs/commands](https://opencode.ai/docs/commands/) (fetched 2026-07-17): markdown files in `~/.config/opencode/commands/` (global) or `.opencode/commands/` (project); filename becomes the command name (`test.md` -> `/test`). Frontmatter fields: `description`, `agent` (which agent executes it), `model` (per-command model override), `subtask` (boolean - force execution as a subagent so it does "not pollute your primary context"), with the body as the prompt template. Argument passing: `$ARGUMENTS` (all args) and positional `$1`, `$2`, ... Shell interpolation via `` !`command` `` (runs in project root, output injected into the prompt) and file inclusion via `@path/to/file`. Commands can equivalently be defined in `opencode.json` under the `command` object with the same properties. Named/keyword arguments beyond positional parameters are not documented - absent.

### 1.3 Custom agents

Per [docs/agents](https://opencode.ai/docs/agents/) (fetched 2026-07-17): markdown files in `~/.config/opencode/agents/` (global) or `.opencode/agents/` (project); "the markdown file name becomes the agent name". Frontmatter: `description` (required - drives automatic delegation), `mode` (`primary` | `subagent` | `all`), `model` (`provider/model-id` override), `temperature`, `permission` (per-tool `allow`/`ask`/`deny`, including glob-scoped bash rules like `"git push": "deny"`), `disable`, `hidden` (hide a subagent from @ autocomplete), `steps` (max agentic iterations), with the body as the system prompt. The authoritative schema ([packages/core/src/v1/config/agent.ts](https://github.com/sst/opencode/blob/dev/packages/core/src/v1/config/agent.ts), fetched 2026-07-17) confirms these plus `top_p`, `variant`, `color`, `options`, and marks `tools: {name: bool}` as "@deprecated Use 'permission' field instead" (still accepted, normalized into permissions). Agents can also be defined in `opencode.json` under `agent`, with `prompt` supporting `{file:./prompts/x.txt}` substitution. Built-ins: Build and Plan (primary), General, Explore, Scout (subagents), plus hidden system agents (Compaction, Title, Summary).

### 1.4 Agent Skills (SKILL.md) - native support

Per [docs/skills](https://opencode.ai/docs/skills/) (fetched 2026-07-17): opencode natively supports the Agent Skills format. Recognized frontmatter: `name` (required), `description` (required), `license`, `compatibility`, `metadata` - unknown fields ignored. Discovery paths, project-local (walking up from cwd to the git worktree root): `.opencode/skills/<name>/SKILL.md`, `.claude/skills/<name>/SKILL.md`, `.agents/skills/<name>/SKILL.md`; global: `~/.config/opencode/skills/`, `~/.claude/skills/`, `~/.agents/skills/`. Skills surface through a native `skill` tool - names and descriptions appear in the tool description and the agent calls `skill({ name: "git-release" })` to load full content (the standard progressive-disclosure shape). Per-skill permissions via `permission.skill` patterns (`"internal-*": "deny"`). Skill names must match `^[a-z0-9]+(-[a-z0-9]+)*$` and equal the directory name. Practical consequence for forge: skills already installed for Claude Code under `.claude/skills/` or `~/.claude/skills/` are picked up by opencode with zero copying.

## 2. Subagents and programmatic spawning

**Subagent spawning: yes, three ways.** Per [docs/agents](https://opencode.ai/docs/agents/): (1) automatic - primary agents delegate to subagents based on their `description`; (2) manual - "@ mentioning" a subagent in a message (`@general help me search...`); (3) programmatic - "other agents invoke them programmatically via the Task tool (controlled by `permission.task`)". Subagent runs are real child sessions the user can navigate into (keybinds `session_child_first`, `session_child_cycle`, `session_parent`).

**Per-task model choice: yes.** The agent definition's `model` frontmatter overrides the global model per agent ([docs/agents](https://opencode.ai/docs/agents/), schema confirmed in [agent.ts](https://github.com/sst/opencode/blob/dev/packages/core/src/v1/config/agent.ts)), and commands carry their own `model` override too ([docs/commands](https://opencode.ai/docs/commands/)). A forge stage brief can therefore pin its stage model in the agent file, as it does in Claude Code.

**Server-level spawning: yes.** Per [docs/server](https://opencode.ai/docs/server/) (fetched 2026-07-17): `opencode serve` exposes an OpenAPI 3.1 HTTP API; `POST /session` accepts `{ parentID?, title? }` - "use `POST /session` with `parentID` to establish parent-child relationships programmatically" - and `GET /session/:id/children` lists a session's children. `POST /session/:id/message` sends a prompt and waits; `prompt_async`, `fork`, `abort`, `command` endpoints exist; `GET /event` is the SSE bus. The typed client is [`@opencode-ai/sdk`](https://opencode.ai/docs/sdk/) (fetched 2026-07-17): `session.create({ body })`, `session.children({ path })`, `session.prompt({ path, body })` with `body.model: { providerID, modelID }` for a per-prompt model override, and `body.noReply: true` to inject context without triggering a reply - a notable primitive for an orchestrator seeding a child session.

## 3. Plugins and events

Per [docs/plugins](https://opencode.ai/docs/plugins/) (fetched 2026-07-17): plugins are JS/TS modules loaded from `~/.config/opencode/plugins/` (global), `.opencode/plugins/` (project), or npm package names listed in the `plugin` array of `opencode.json` (e.g. `"plugin": ["opencode-helicone-session", "@my-org/custom-plugin"]`); directory files "are automatically loaded at startup". A plugin exports an async function receiving `{ project, client, $, directory, worktree }` - `client` is a full SDK client bound to the running server, `$` is Bun's shell - and returns a hooks object.

The authoritative `Hooks` interface ([packages/plugin/src/index.ts](https://github.com/sst/opencode/blob/dev/packages/plugin/src/index.ts), `dev` branch, fetched 2026-07-17) contains: `dispose`, `event` (catch-all bus subscriber), `config`, `tool` (custom tool map), `auth`, `provider`, `chat.message`, `chat.params`, `chat.headers`, `permission.ask`, `command.execute.before`, `tool.execute.before`, `tool.execute.after`, `tool.definition`, `shell.env`, and experimental hooks `experimental.chat.messages.transform`, `experimental.chat.system.transform`, `experimental.provider.small_model`, `experimental.session.compacting`, `experimental.compaction.autocontinue`, `experimental.text.complete`. Bus events reachable through the `event` hook include ([docs/plugins](https://opencode.ai/docs/plugins/)): `session.created`, `session.idle`, `session.error`, `session.status`, `session.updated`, `session.compacted`, `session.deleted`, `session.diff`, `message.updated`, `message.removed`, `message.part.updated`, `message.part.removed`, `permission.asked`, `permission.replied`, `file.edited`, `file.watcher.updated`, `command.executed`, `lsp.client.diagnostics`, `todo.updated`, `tui.prompt.append`, `tui.command.execute`, `tui.toast.show`, `server.connected`, `installation.updated`.

**(a) Context injection at session start: yes, several channels.** `chat.message` fires on every new user message with a mutable `output: { message: UserMessage; parts: Part[] }` ([source](https://github.com/sst/opencode/blob/dev/packages/plugin/src/index.ts), lines 234-243) - a plugin can append or rewrite parts, i.e. inject context deterministically per message. `experimental.chat.system.transform` modifies the system prompt string; `experimental.chat.messages.transform` rewrites the whole message array ([source](https://github.com/sst/opencode/blob/dev/packages/plugin/src/index.ts)). The `config` hook receives the loaded config at startup - the runtime comment reads "Notify plugins of current config" ([packages/opencode/src/plugin/index.ts](https://github.com/sst/opencode/blob/dev/packages/opencode/src/plugin/index.ts)); it is a notification, and config mutation from it is not a documented contract.

**(b) Blocking/gating the end of a session: IMPOSSIBLE.** There is no Stop-hook equivalent anywhere in the surface. The `Hooks` interface has no `session.*` hook at all - session lifecycle is observable only through the fire-and-forget `event` bus, and `session.idle` (the "session finished responding" signal, shown in the docs' notification example) cannot veto, delay, or return feedback into the turn ([docs/plugins](https://opencode.ai/docs/plugins/), [source](https://github.com/sst/opencode/blob/dev/packages/plugin/src/index.ts)). What CAN block: `tool.execute.before` may `throw` to abort a tool call (docs example: throwing on reads of `.env` files), `permission.ask` may set `output.status = "deny"`, and `command.execute.before` intercepts command runs. The closest Stop-gate emulation is reactive: a plugin listens for `session.idle` on the bus and uses its SDK `client` to `session.prompt` the same session with review instructions - a post-hoc nudge the model has already stopped before, not a gate, and it burns a fresh turn. The legacy `experimental.hook` config keys (`session_completed`, `file_edited`) that early-2025 versions carried are gone: GitHub code search over sst/opencode returns zero hits for either identifier (searched 2026-07-17); that mechanism no longer exists.

**(c) Custom tools: yes, two ways.** Plugins return a `tool` map built with the `tool()` helper (`description`, Zod-style `args` via `tool.schema`, `execute(args, context)`); "the plugin tool takes precedence" over same-named built-ins ([docs/plugins](https://opencode.ai/docs/plugins/)). Standalone tool files also work without a plugin: `.opencode/tools/` (project) or `~/.config/opencode/tools/` (global), filename becomes the tool name, `execute` context carries `agent, sessionID, messageID, directory, worktree` ([docs/custom-tools](https://opencode.ai/docs/custom-tools/), fetched 2026-07-17). `tool.definition` additionally lets a plugin rewrite built-in tool descriptions/parameters as sent to the LLM ([source](https://github.com/sst/opencode/blob/dev/packages/plugin/src/index.ts)).

## 4. Model selection

Per [docs/models](https://opencode.ai/docs/models/) (fetched 2026-07-17): opencode uses "the AI SDK and Models.dev to support 75+ LLM providers"; model ids are `provider_id/model_id` (e.g. `opencode/gpt-5.1-codex`, or a custom provider key plus model id). Config keys: `model` (primary) and `small_model` (lightweight tasks like titles/summaries) ([docs/config](https://opencode.ai/docs/config/), fetched 2026-07-17). Resolution order at startup: `--model`/`-m` flag, config `model`, last used model, internal defaults. Providers authenticate via the `/connect` command (docs also reference `opencode auth login` historically); `/models` lists what is available. Per-agent `model` frontmatter and per-command `model` frontmatter both override the global choice (sections 1.2, 1.3). [OpenCode Zen](https://opencode.ai/docs/zen/) (fetched 2026-07-17) is the team's own gateway - "a list of tested and verified models provided by the opencode team", pay-as-you-go, model ids `opencode/<model-id>` - an optional provider, not a requirement. For forge this means the multi-model stage dispatch (cheap lens models, strong architect models) maps directly onto agent-level `model` fields, with a far wider provider pool than Claude Code.

## 5. Distribution

**There is no plugin marketplace.** Nothing like Claude Code's `claude-plugins-official`/`claude-community` marketplaces exists; the [docs/ecosystem](https://opencode.ai/docs/ecosystem/) page (fetched 2026-07-17) is a hand-curated directory of ~30 community plugins and ~11 projects, distribution is "direct GitHub repositories", and the external indexes it points at (awesome-opencode repo, opencode.cafe - discovery indexes, not registries) are community-run. Stated plainly: no install counts, no review pipeline, no one-command marketplace add.

The actual channels, per surface type:

- **JS plugins**: npm. List the package name in the `plugin` array of `opencode.json` and opencode resolves it at startup ([docs/plugins](https://opencode.ai/docs/plugins/)); the config schema accepts a string or a `[name, options]` tuple ([packages/core/src/v1/config/plugin.ts](https://github.com/sst/opencode/blob/dev/packages/core/src/v1/config/plugin.ts)). Updates propagate through npm version resolution. Local plugins are plain files in `.opencode/plugins/` or `~/.config/opencode/plugins/`.
- **Agents, commands, skills, tools, instructions**: plain files. Global under `~/.config/opencode/{agents,commands,skills,tools}/`, per-project under `.opencode/{agents,commands,skills,tools}/` (sections 1.2-1.4, 3c). No installer, no registry - copying (or a repo the user clones/symlinks) is the mechanism. No update channel exists for copied files; a forge adapter must ship its own install/update script or lean on git.
- **Config**: `opencode.json` in the project root is version-controllable and merges over global config; configs "merge rather than replace" across a documented precedence chain (remote `.well-known/opencode`, global, `OPENCODE_CONFIG`, project, `.opencode` dirs, inline env, managed) ([docs/config](https://opencode.ai/docs/config/)). A committed `opencode.json` carrying `instructions`, `agent`, `command`, and `plugin` entries is the closest thing to a one-file forge manifest.
- **The Claude-compat shortcut**: because opencode reads `.claude/skills/` and `~/.claude/skills/` natively ([docs/skills](https://opencode.ai/docs/skills/)) and falls back to `CLAUDE.md`/`~/.claude/CLAUDE.md` ([docs/rules](https://opencode.ai/docs/rules/)), a forge installation done for Claude Code already surfaces its skills and rules in opencode with no adapter work at all.
- **opencode itself** updates via the `autoupdate` config key (`true`/`false`/`"notify"`) ([docs/config](https://opencode.ai/docs/config/)).

## 6. What is impossible / gaps vs Claude Code

- **No Stop gate - IMPOSSIBLE.** Nothing can block or gate the end of a session or force continuation with feedback, the way Claude Code's Stop hook returns a blocking reason and re-engages the model. `session.idle` is notify-only; the `Hooks` interface has no session-lifecycle interception point ([packages/plugin/src/index.ts](https://github.com/sst/opencode/blob/dev/packages/plugin/src/index.ts)). Forge's review-gate enforcement cannot be ported at the same tier; the ceiling is (a) tool-level blocking (`tool.execute.before` throw, `permission.ask` deny) and (b) a reactive `session.idle` -> `client.session.prompt` re-injection loop, which the model experiences as a new user turn, not a refusal to stop.
- **No hook-based deterministic prompt-time enforcement of arbitrary policy at turn boundaries.** There is no UserPromptSubmit/PreCompact/SessionStart/SessionEnd hook family; the equivalents that exist (`chat.message`, `experimental.chat.system.transform`, `experimental.session.compacting`) cover message/system mutation and compaction but there is no exit-code-blocks-with-stderr contract anywhere - blocking is exception-throwing in `tool.execute.before` only.
- **No marketplace, no install counts, no one-command plugin install with bundled skills+agents+hooks.** Claude Code's plugin unit (skills + agents + hooks + MCP + settings in one marketplace install) has no opencode equivalent; a forge port must split into an npm plugin (JS enforcement/tools) plus copied markdown files plus a committed `opencode.json`, with self-managed updates ([docs/ecosystem](https://opencode.ai/docs/ecosystem/), [docs/plugins](https://opencode.ai/docs/plugins/)).
- **Hooks are JS/TS only.** Claude Code hooks are arbitrary shell commands; opencode plugin hooks must be JavaScript/TypeScript modules (custom tools may shell out, but the hook surface itself is JS) ([docs/plugins](https://opencode.ai/docs/plugins/), [docs/custom-tools](https://opencode.ai/docs/custom-tools/)).
- **Not gaps**: subagents with per-agent models, programmatic child sessions, SKILL.md skills, per-command models, tool blocking, permission deny, system-prompt and message transforms, and multi-provider model dispatch are all present and in most cases broader than Claude Code's equivalents.
