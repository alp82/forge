# Codex CLI extension surface survey

Date: 2026-07-17. Question: what can a forge adapter build on in OpenAI Codex CLI - instruction surface, subagents, hooks, model selection, packaging - and what is impossible there? Wayfinder ticket #32.

Primary sources only, all fetched 2026-07-17: the official Codex docs (canonical host `developers.openai.com/codex`, which 308-redirects to `learn.chatgpt.com` - both URL forms below resolve to the same pages), the [openai/codex](https://github.com/openai/codex) GitHub repo (README, docs/, [releases](https://github.com/openai/codex/releases)), and [agentskills.io](https://agentskills.io). The repo's docs/ folder (config.md, skills.md, slash_commands.md, exec.md, agents_md.md) has been reduced to stubs that point at the hosted docs, so the hosted docs are the authoritative surface; version anchoring comes from the GitHub releases page and the hosted [changelog](https://learn.chatgpt.com/codex/changelog). Current release: **Codex CLI 0.144.5, released 2026-07-16** ([releases](https://github.com/openai/codex/releases)); 0.145.0 alphas are streaming daily. One secondary source is used for a single dating claim and labeled as such. House style: hyphens only.

## Executive summary

Codex CLI in mid-2026 is a far richer adapter target than the 2025-era CLI: it now has native parallel subagents with per-agent model override, a full lifecycle hook system whose `Stop` hook can return `decision: "block"` - a mechanical Stop-gate equivalent - and a plugin format (`.codex-plugin/plugin.json`) that bundles skills plus hooks and installs from git, npm, or local marketplaces without OpenAI review. Skills follow the open Agent Skills standard (SKILL.md folders under `.agents/skills/`, the same layout Claude Code consumes), so forge's briefs port as files; custom prompts (`~/.codex/prompts`, `$1..$9`/`$ARGUMENTS`) still work but are "deprecated in favor of skills". The load-bearing caveats: hooks are gated behind a `features.hooks` config flag and carry no published stable-maturity label, the `notify` hook is explicitly fire-and-forget (only the hook system can block), subagent nesting defaults to depth 1, and Codex-as-MCP-server is absent from the current MCP docs. Every forge primitive - skills, a review Stop-gate, multi-model dispatch, plugin distribution - has a mechanical home on Codex today.

## 1. Instruction surface

**AGENTS.md.** Codex builds its instruction chain from a three-tier hierarchy: global (`~/.codex/AGENTS.override.md` first, then `~/.codex/AGENTS.md`), then project scope, walking "from Git root down to current directory" checking `AGENTS.override.md`, `AGENTS.md`, then configured fallback names at each level ([AGENTS.md docs](https://learn.chatgpt.com/codex/agent-configuration/agents-md)). Files concatenate root-downward: "Files closer to your current directory override earlier guidance because they appear later in the combined prompt". Size cap is `project_doc_max_bytes`, default **32 KiB**, adjustable in `~/.codex/config.toml`; "Codex skips empty files and stops adding files once the combined size reaches the limit". `project_doc_fallback_filenames` adds alternate names (e.g. `["TEAM_GUIDE.md", ".agents.md"]`); Codex "rebuilds the instruction chain on every run". `model_instructions_file` can replace the built-in instructions wholesale, and `CODEX_HOME` redirects the whole config home per invocation ([config reference](https://learn.chatgpt.com/codex/config-file/config-reference), [AGENTS.md docs](https://learn.chatgpt.com/codex/agent-configuration/agents-md)).

**Custom prompts.** Still present but explicitly "deprecated in favor of skills for reusable instructions" ([custom prompts](https://learn.chatgpt.com/codex/custom-prompts)). Markdown files in `~/.codex/prompts/` with YAML frontmatter (`description`, `argument-hint`), invoked as `/prompts:<name>`. Arguments: positional "`$1` through `$9` for space-separated values, or `$ARGUMENTS` for all", plus named uppercase placeholders (`$FILE`, `$TICKET_ID`) supplied as `KEY=value`, and `$$` for a literal dollar sign. The docs contrast them with skills: "Custom prompts require explicit invocation and aren't shared via repositories." The built-in slash command set (`/model`, `/permissions`, `/plan`, `/review`, `/status`, `/diff`, etc.) is documented separately ([developer commands](https://learn.chatgpt.com/docs/developer-commands?surface=cli)).

**Agent Skills.** Codex supports SKILL.md folders per the open standard; [agentskills.io](https://agentskills.io) lists OpenAI Codex as an adopting client with instructions at `developers.openai.com/codex/skills`. Discovery scopes, in order: `$CWD/.agents/skills`, `$REPO_ROOT/.agents/skills`, `$HOME/.agents/skills`, `/etc/codex/skills` (admin), plus built-in OpenAI-provided skills ([build skills](https://learn.chatgpt.com/codex/build-skills)). A skill is a folder with `SKILL.md` (required frontmatter: `name`, `description`) plus optional `scripts/`, `references/`, `assets/`, and an optional `agents/openai.yaml` for appearance and dependencies, including `allow_implicit_invocation` (default true). Invocation is "Explicit: directly mention via `/skills` or `$skillname`" or "Implicit: Codex automatically selects skills matching task descriptions". A built-in `$skill-installer` skill installs skills by name. Skills support landed in December 2025 (dated via [Simon Willison](https://simonw.substack.com/p/openai-are-quietly-adopting-skills), secondary, corroborated by the standard's client list); skill machinery is still actively evolving in the current line, e.g. "Emit implicit skill usage" in 0.143.0, 2026-07-08 ([changelog](https://learn.chatgpt.com/codex/changelog)).

## 2. Subagent spawning

Native subagents exist. "Work mode and Codex can run subagent workflows by spawning specialized agents in parallel and then collecting their results in one response", and "Current Codex releases enable subagent workflows by default" ([subagents](https://learn.chatgpt.com/codex/agent-configuration/subagents)). Two configuration mechanisms:

- **Custom agent files**: standalone TOML files in `~/.codex/agents/` (personal) or `.codex/agents/` (project-scoped). "Every standalone custom agent file must define: `name`, `description`, `developer_instructions`" - note TOML, not the markdown-frontmatter agent format Claude Code uses. Each agent file can override "`model` and `model_reasoning_effort`" independently of the parent session, so **per-task model choice is native** ([subagents](https://learn.chatgpt.com/codex/agent-configuration/subagents)).
- **Global `[agents]` config**: `agents.max_threads` (default 6 concurrent subagent threads) and `agents.max_depth` (default 1 - subagents do not spawn subagents unless raised) ([subagents](https://learn.chatgpt.com/codex/agent-configuration/subagents)).

CSV-driven batch spawning (`spawn_agents_on_csv`) is "explicitly marked as experimental". The models page also exposes an "Ultra" reasoning mode that "uses subagents for parallel task handling on complex work" ([models](https://learn.chatgpt.com/codex/models)).

**`codex exec` as self-spawn** remains available and is the scriptable fallback: it "runs Codex from scripts and CI without the interactive TUI, streaming progress to stderr and final output to stdout" ([non-interactive mode](https://learn.chatgpt.com/docs/non-interactive-mode)). Relevant flags: `--json` turns stdout into "a JSON Lines (JSONL) stream so you can capture every event Codex emits" (`thread.started`, `turn.started`, `turn.completed`, `turn.failed`, `item.*`); `--output-schema <path>` requires the final response to conform to a JSON Schema; `-o`/`--output-last-message <path>` writes the final message to a file; `-m` selects the model per invocation (`codex exec -m gpt-5.6`, [models](https://learn.chatgpt.com/codex/models)); `--ephemeral` skips session persistence; `resume <SESSION_ID>` continues a run; stdin works as context (`cmd | codex exec "instruction"`) or as the whole prompt (`codex exec -`); `--sandbox workspace-write | danger-full-access`, `--ignore-user-config`, `--ignore-rules`, `--skip-git-repo-check` control the environment ([non-interactive mode](https://learn.chatgpt.com/docs/non-interactive-mode)).

## 3. Hooks and lifecycle events

Codex now ships a full lifecycle hook system, configured in `hooks.json` (`~/.codex/hooks.json` or `<repo>/.codex/hooks.json`) or inline `[hooks]` tables in `config.toml`, and **gated behind `features.hooks`**: "Enables lifecycle hooks from `hooks.json` or inline `[hooks]` config" ([config reference](https://learn.chatgpt.com/codex/config-file/config-reference), [advanced config](https://learn.chatgpt.com/codex/config-file/config-advanced)). Events ([hooks](https://learn.chatgpt.com/codex/hooks)):

- Turn-scoped: `PreToolUse`, `PermissionRequest`, `PostToolUse`, `PreCompact`, `PostCompact`, `UserPromptSubmit`, `SubagentStop`, `Stop`.
- Session-scoped: `SessionStart` (init or resume), `SubagentStart`.

Hook entries take a `matcher` regex plus `command`, `timeout`, `statusMessage`; hooks receive a JSON object on stdin (`session_id`, `cwd`, `hook_event_name`, `permission_mode`) and respond via exit code or structured JSON on stdout. "Exit 0 with no output" is success; "Exit code 2: indicates blocking/denial with reason written to stderr" ([hooks](https://learn.chatgpt.com/codex/hooks)).

**Blocking is real.** `PreToolUse` denies a tool call with `permissionDecision: "deny"`; `PermissionRequest` can "deny the request"; `PostToolUse` can "replace the tool result with your feedback" and halt via `continue: false`; `UserPromptSubmit` supports `decision: "block"`; and critically **`Stop`/`SubagentStop` return `decision: "block"` to force additional processing passes** - a mechanical Stop-gate equivalent to Claude Code's Stop hook ([hooks](https://learn.chatgpt.com/codex/hooks)). Admin governance exists: `allow_managed_hooks_only = true` "ignore[s] user, project, and session hook configs while still allowing managed hooks" ([repo config.md](https://github.com/openai/codex/blob/main/docs/config.md)).

**`notify` cannot block.** The older `notify` config option is a separate, simpler channel: `notify = ["python3", "/path/to/notify.py"]` receives "a JSON payload from Codex" for `agent-turn-complete` events (`type`, `thread-id`, `turn-id`, `cwd`, `input-messages`, `last-assistant-message`) and "operates as fire-and-forget... asynchronously... without blocking the main workflow" ([advanced config](https://learn.chatgpt.com/codex/config-file/config-advanced), [config reference](https://learn.chatgpt.com/codex/config-file/config-reference)). Use hooks, not notify, for gating.

**Context injection at session start** is served by the AGENTS.md chain (section 1), `model_instructions_file`, and the `SessionStart` hook event; hook development is recent and ongoing (e.g. "trace hook command execution" and hook prompt items in 0.143.0, 2026-07-08, [changelog](https://learn.chatgpt.com/codex/changelog)). Tool-approval behavior is separately governed by `approval_policy` (`untrusted | on-request | never`), `sandbox_mode` (`read-only | workspace-write | danger-full-access`), and `approvals_reviewer` (`user | auto_review`) ([config reference](https://learn.chatgpt.com/codex/config-file/config-reference)).

## 4. Model selection

Selection channels: `model = "gpt-5.6"` in `config.toml`; per-invocation `-m/--model` on both `codex` and `codex exec`; `/model` in the TUI; and profiles - "Profiles let you save named configuration layers and switch between them from the CLI" via `--profile <name>`, which loads `~/.codex/<name>.config.toml` as an overlay ([models](https://learn.chatgpt.com/codex/models), [advanced config](https://learn.chatgpt.com/codex/config-file/config-advanced)). Custom endpoints go through `model_providers.<id>` plus `model_provider` (default `openai`), and `model_catalog_json` loads a model catalog at startup ([config reference](https://learn.chatgpt.com/codex/config-file/config-reference)).

Current model lineup ([models](https://learn.chatgpt.com/codex/models)): the GPT-5.6 family - `gpt-5.6-sol` ("Flagship GPT-5.6 model with the strongest capability for complex coding, computer use, research, and cybersecurity"), `gpt-5.6-terra` ("Balanced GPT-5.6 model for everyday work"), `gpt-5.6-luna` ("Fast and affordable GPT-5.6 model that delivers strong capability at the lowest cost") - plus `gpt-5.5`, `gpt-5.4`, `gpt-5.4-mini`, and `gpt-5.3-codex-spark` (text-only, ChatGPT Pro). Default: "GPT-5.6-Sol with medium reasoning effort". Reasoning effort config values are `minimal | low | medium | high | xhigh` ([config reference](https://learn.chatgpt.com/codex/config-file/config-reference)); the models page surfaces these as Low through "Max" plus the subagent-powered "Ultra" mode. Subagent files override `model` and `model_reasoning_effort` per agent (section 2), so a forge adapter gets per-stage model dispatch natively.

## 5. Packaging and distribution

**Codex itself**: `npm install -g @openai/codex`, `brew install --cask codex`, or GitHub release binaries for macOS and Linux ([README](https://github.com/openai/codex)).

**Third-party instruction sets** now have a first-class plugin format, not just copied files. A plugin's entry point is `.codex-plugin/plugin.json` ("name", "version", and "description" identify the plugin) referencing bundled components by relative path: `skills/` for skills, `hooks/hooks.json` for lifecycle hooks, `.mcp.json` for MCP servers, `.app.json` for apps, `assets/` for icons ([build plugins](https://learn.chatgpt.com/codex/build-plugins)). Distribution channels:

- **Self-serve marketplaces**: repo-level `$REPO_ROOT/.agents/plugins/marketplace.json`, personal `~/.agents/plugins/marketplace.json`, plus "Git-backed sources, npm registry, or local paths", managed via `codex plugin marketplace add [source]` / `codex plugin marketplace list`. "Remote plugins are now enabled by default, with richer catalog rows, npm marketplace sources, and visible remote/local versions" (0.143.0, 2026-07-08, [changelog](https://learn.chatgpt.com/codex/changelog)). No OpenAI review required on this path.
- **Public Plugins Directory**: submission "through the plugin submission portal" at platform.openai.com/plugins; "OpenAI reviews the submission"; "After publication, the plugin appears in the Plugins Directory available in both ChatGPT and Codex". Requires verified developer identity, test cases, and policy URLs ([submit plugins](https://learn.chatgpt.com/codex/submit-plugins)). Directory plugins come in three shapes: "A skills-only plugin", "An app-only plugin", "An app-plus-skills plugin".
- **Bare skills**: plain SKILL.md folders dropped into any discovery scope (section 1), or installed via the built-in `$skill-installer` skill ([build skills](https://learn.chatgpt.com/codex/build-skills)).

**MCP, briefly.** Codex is a full MCP client: `codex mcp add <server-name> -- <command>` or `mcp_servers` in `~/.codex/config.toml` / project `.codex/config.toml`; "STDIO servers" and "Streamable HTTP servers"; bearer, OAuth, and ChatGPT session auth; per-server `enabled_tools`/`disabled_tools` and per-tool `approval_mode` (auto, prompt, writes, approve). "The ChatGPT desktop app, Codex CLI, and IDE extension support MCP servers and share MCP configuration for the same Codex host" ([MCP](https://learn.chatgpt.com/codex/extend/mcp)). The current MCP page documents Codex only as a client; a `codex mcp serve` server mode is not described there.

## 6. What is impossible

- **Blocking via `notify` is impossible.** It "operates as fire-and-forget... without blocking the main workflow" ([advanced config](https://learn.chatgpt.com/codex/config-file/config-advanced)). Any gate must go through the hooks system.
- **Hooks without opt-in are impossible.** Lifecycle hooks require the `features.hooks` flag ([config reference](https://learn.chatgpt.com/codex/config-file/config-reference)), and no published maturity label marks them stable - the [feature maturity](https://learn.chatgpt.com/codex/feature-maturity) page defines the ladder ("Under development", "Experimental", "Beta", "Stable") without pinning these features to a rung. An adapter cannot assume hooks are on in a consumer's install; it must enable them or degrade.
- **Deep subagent trees are off by default.** `agents.max_depth` defaults to 1 ([subagents](https://learn.chatgpt.com/codex/agent-configuration/subagents)); an orchestrator-of-orchestrators needs explicit config.
- **Custom prompts are a dead surface.** "Deprecated in favor of skills" ([custom prompts](https://learn.chatgpt.com/codex/custom-prompts)) - building adapter machinery on `~/.codex/prompts` argument expansion invites removal.
- **Codex-as-MCP-server is undocumented in current docs.** The [MCP page](https://learn.chatgpt.com/codex/extend/mcp) covers only the client role; an adapter should not depend on serving Codex to another agent over MCP.
- **Markdown agent definitions are impossible.** Custom subagents are TOML files requiring `name`, `description`, `developer_instructions` ([subagents](https://learn.chatgpt.com/codex/agent-configuration/subagents)) - Claude Code agent markdown does not port as-is.
- **Unreviewed public-directory listing is impossible.** The Plugins Directory path runs through OpenAI identity verification and review ([submit plugins](https://learn.chatgpt.com/codex/submit-plugins)); only git/npm/local marketplaces are self-serve.

## 7. Implications for a forge adapter

Three decisions the survey feeds:

1. **Subagent spawning: available as an adapter capability.** Native parallel subagents with per-agent `model` and `model_reasoning_effort` override cover forge's stage-dispatch model directly; `codex exec --json --output-schema -m <model>` remains as the structured-return self-spawn for anything the native `[agents]` surface cannot express. The depth-1 default confines forge to a flat orchestrator-plus-workers shape unless the adapter raises `agents.max_depth`.
2. **Enforcement tier: mechanical.** The `Stop` hook's `decision: "block"` (and `SubagentStop`) gives Codex a true Stop-gate - forge's review wave can be mechanically enforced, not prose-only. This reverses the 2025-era finding in [landscape-survey.md](landscape-survey.md) that enforcement had "no widely adopted analog" outside Claude Code. Caveats to engineer around: the `features.hooks` opt-in, unlabeled maturity, and admin `allow_managed_hooks_only` environments where user hooks are ignored.
3. **Distribution: ship a Codex plugin.** `.codex-plugin/plugin.json` bundling `skills/` (forge's briefs, already SKILL.md-shaped) plus `hooks/hooks.json` (the Stop-gate) mirrors forge's Claude Code plugin one-to-one, installable self-serve from a git or npm marketplace source via `codex plugin marketplace add`. A skills-only fallback - plain folders under `.agents/skills/` - keeps the briefs portable to consumers who never enable plugins or hooks, at the cost of dropping to prose-only enforcement.
