# Gemini CLI extension surface survey

Date: 2026-07-17. Question: what can a forge adapter build on in Google's Gemini CLI - instruction surface, subagent spawning, hooks, model selection, and distribution? Wayfinder ticket #33.

Primary sources only: the `docs/` tree of [google-gemini/gemini-cli](https://github.com/google-gemini/gemini-cli) fetched from the `main` branch on 2026-07-17 (current stable release v0.51.0, 2026-07-16, per the [releases page](https://github.com/google-gemini/gemini-cli/releases)), plus the CLI reference in the same tree. House style: hyphens only. The survey feeds three decisions: (a) whether subagent spawning is a required adapter capability, (b) what enforcement tier Gemini CLI can support, and (c) how to distribute forge there.

## Executive summary

Gemini CLI is a far closer match to forge's Claude Code substrate than expected - all three decision axes come back positive. **Spawn:** subagents exist as markdown-plus-YAML-frontmatter files in `.gemini/agents/` with isolated context loops, and the frontmatter carries a per-agent `model` field, so per-task model choice is real ([docs/core/subagents.md](https://github.com/google-gemini/gemini-cli/blob/main/docs/core/subagents.md), fetched 2026-07-17) - the caveats are that the feature is flagged experimental (on by default, `experimental.enableAgents`), subagents cannot nest, and each instance is single-threaded. **Enforcement:** the hooks system is Claude Code-grade or better - eleven events spanning tool, agent, and model lifecycles, configured in `settings.json` or shipped in an extension's `hooks/hooks.json`, speaking JSON over stdin/stdout with exit-code-2 blocking; critically, the `AfterAgent` event supports `"decision": "block"` which "triggers an automatic retry turn", a direct Stop-gate equivalent that can hold the agent until tests pass ([docs/hooks/reference.md](https://github.com/google-gemini/gemini-cli/blob/main/docs/hooks/reference.md), [docs/hooks/writing-hooks.md](https://github.com/google-gemini/gemini-cli/blob/main/docs/hooks/writing-hooks.md), both fetched 2026-07-17). **Distribution:** extensions are the plugin analog - one `gemini-extension.json` manifest bundling context files, TOML commands, MCP servers, skills, hooks, and subagents - installed straight from a Git URL with `gemini extensions install`, updated with `gemini extensions update --all` or `--auto-update`, and listed in a browsable gallery at [geminicli.com/extensions/browse/](https://geminicli.com/extensions/browse/) ([docs/extensions/index.md](https://github.com/google-gemini/gemini-cli/blob/main/docs/extensions/index.md), [docs/cli/cli-reference.md](https://github.com/google-gemini/gemini-cli/blob/main/docs/cli/cli-reference.md), fetched 2026-07-17). Gemini CLI also adopted the Agent Skills standard (`SKILL.md`, `.agents/skills/` alias "compatible with other AI agent tools following the Agent Skills standard"), so forge's skill briefs port with near-zero translation. The gaps that remain are narrow: no per-command model override in the TOML command format, no subagent nesting, and the whole subagent surface sits behind an experimental flag Google can still move.

## 1. Instruction and skill surface

Four instruction channels exist, all plain files.

**Context files (GEMINI.md).** Hierarchical: a global `~/.gemini/GEMINI.md`, then workspace context files discovered from the project directory and its parents, then just-in-time context files auto-scanned when tools touch a location. All found files are concatenated and "sent to the model with every prompt". Modular imports use `@file.md` syntax supporting relative and absolute paths. The filename is configurable via `context.fileName` in `settings.json` and accepts a list (the doc's example: `["AGENTS.md", "CONTEXT.md", "GEMINI.md"]`). `/memory show` and `/memory reload` manage the loaded set. ([docs/cli/gemini-md.md](https://github.com/google-gemini/gemini-cli/blob/main/docs/cli/gemini-md.md), fetched 2026-07-17)

**Custom commands.** TOML files, one command per file; required field `prompt`, optional `description`. Locations: `~/.gemini/commands/` (global) and `<project>/.gemini/commands/` (project wins on name clash). Namespacing comes from the file path - `git/commit.toml` becomes `/git:commit`. Argument passing: `{{args}}` placeholder (raw outside shell blocks, shell-escaped inside); if absent, arguments are appended to the prompt. `!{...}` executes shell commands and injects the output (with a security check and user confirmation); `@{...}` embeds file or directory content, respecting `.gitignore` and supporting multimodal files. ([docs/cli/custom-commands.md](https://github.com/google-gemini/gemini-cli/blob/main/docs/cli/custom-commands.md), fetched 2026-07-17)

**Agent skills.** Gemini CLI implements the Agent Skills open standard: a skill is "a self-contained directory that packages instructions and assets into a discoverable capability" with a `SKILL.md` whose required frontmatter is exactly `name` and `description` ([docs/cli/creating-skills.md](https://github.com/google-gemini/gemini-cli/blob/main/docs/cli/creating-skills.md), fetched 2026-07-17). Discovery tiers, lowest to highest precedence: built-in, extension-shipped, user (`~/.gemini/skills/` or the `~/.agents/skills/` alias), workspace (`.gemini/skills/` or `.agents/skills/`); the `.agents/skills/` alias is explicitly "compatible with other AI agent tools following the Agent Skills standard" ([docs/cli/skills.md](https://github.com/google-gemini/gemini-cli/blob/main/docs/cli/skills.md), fetched 2026-07-17). Lifecycle: name+description injected at session start, model calls an `activate_skill` tool, user consents, SKILL.md body and folder access are injected. A `gemini skills` command group manages them: `list`, `install <source>` (Git, path, or file), `link` (symlink for development), `uninstall`, `enable`/`disable` ([docs/cli/cli-reference.md](https://github.com/google-gemini/gemini-cli/blob/main/docs/cli/cli-reference.md), fetched 2026-07-17).

**Extensions.** The bundling unit - see section 5.

## 2. Subagent spawning

Subagents exist and match the Claude Code agent shape closely ([docs/core/subagents.md](https://github.com/google-gemini/gemini-cli/blob/main/docs/core/subagents.md), fetched 2026-07-17):

- **Isolated context:** "Interactions with a subagent happen in a separate context loop, which saves tokens in your main conversation history." Each subagent has its own system prompt, persona, and restricted toolset.
- **Format:** markdown files with YAML frontmatter in `.gemini/agents/*.md` (project) or `~/.gemini/agents/*.md` (user); extensions ship them in an `agents/` directory ([docs/extensions/reference.md](https://github.com/google-gemini/gemini-cli/blob/main/docs/extensions/reference.md), fetched 2026-07-17). Required frontmatter: `name`, `description`. Optional: `model`, `tools`, `temperature`, `max_turns`, `timeout_mins`.
- **Per-task model choice: yes.** The `model` frontmatter field is a "specific model to use (for example, `gemini-3-preview`)", and [docs/cli/model.md](https://github.com/google-gemini/gemini-cli/blob/main/docs/cli/model.md) (fetched 2026-07-17) confirms the inverse: "The `/model` command (and the `--model` flag) does not override the model used by sub-agents" - the agent file owns its model.
- **Invocation:** automatic delegation when a task matches an agent's description, or explicit forcing with `@subagent_name`.
- **Status and limits:** enabled by default but experimental (`experimental.enableAgents` in `settings.json` turns it off). "Subagents cannot call other subagents" - no nesting - and execution is single-threaded per instance.

## 3. Hooks and events

The hooks system ([docs/hooks/reference.md](https://github.com/google-gemini/gemini-cli/blob/main/docs/hooks/reference.md), fetched 2026-07-17) defines eleven events: `BeforeTool`/`AfterTool` (tool lifecycle), `BeforeAgent`/`AfterAgent` (turn lifecycle), `BeforeModel`/`BeforeToolSelection`/`AfterModel` (LLM request lifecycle), and `SessionStart`/`SessionEnd`/`Notification`/`PreCompress` (session lifecycle). Hooks are configured under a `hooks` object in `settings.json` (matcher regex, sequential/parallel, `type: "command"` with a shell command and timeout) or shipped by an extension in `hooks/hooks.json`. Protocol: JSON on stdin, JSON on stdout, stderr for logs; exit code 0 = success, exit code 2 = system block with stderr as the rejection reason, other codes = non-fatal warning.

**Blocking is first-class.** Six events accept a `"decision": "deny"`/`"block"` output: `BeforeTool` (prevents tool execution), `AfterTool` (hides tool output, `reason` replaces it), `BeforeAgent` (blocks the turn), `AfterAgent`, `BeforeModel`, `AfterModel`. A `"continue": false` output halts the agent loop outright.

**The Stop-gate equivalent exists: `AfterAgent`.** It fires "after the model generates the final response", and a `"decision": "block"` from it "triggers an automatic retry turn" - the agent must address the `reason` before its answer stands. [docs/hooks/writing-hooks.md](https://github.com/google-gemini/gemini-cli/blob/main/docs/hooks/writing-hooks.md) (fetched 2026-07-17) shows the exact pattern forge's review gate needs: a validator script that inspects the response and emits `{"decision": "block", "reason": "Your response is missing a Summary section. Please add one."}`. A test-gate hook that shells out to the test runner and blocks on red is directly expressible.

**Context injection.** `SessionStart` fires on startup, resume, or `/clear` but is advisory (no blocking). Documented context injection rides on `BeforeAgent`, whose `hookSpecificOutput.additionalContext` field injects text into the agent's context before it processes the user request - the writing-hooks guide's memory example injects `## Relevant Memories` this way ([docs/hooks/writing-hooks.md](https://github.com/google-gemini/gemini-cli/blob/main/docs/hooks/writing-hooks.md), fetched 2026-07-17). Since `BeforeAgent` fires on every user prompt including the first, session-start context injection is covered.

## 4. Model selection

Selection surfaces: the `--model` flag at startup, the interactive `/model` dialog, and `settings.json` ([docs/cli/model.md](https://github.com/google-gemini/gemini-cli/blob/main/docs/cli/model.md), fetched 2026-07-17). Current model IDs listed in that doc: `gemini-3-pro-preview`, `gemini-3-flash-preview`, `gemini-2.5-pro`, `gemini-2.5-flash`. Two "Auto" options route by task complexity: Auto (Gemini 3) picks between Pro Preview and Flash Preview, Auto (Gemini 2.5) between Pro and Flash; dedicated docs exist for [model routing](https://github.com/google-gemini/gemini-cli/blob/main/docs/cli/model-routing.md). Per-agent override is the subagent `model` frontmatter field (section 2). The custom-command TOML format documents only `prompt` and `description` - no per-command model field ([docs/cli/custom-commands.md](https://github.com/google-gemini/gemini-cli/blob/main/docs/cli/custom-commands.md), fetched 2026-07-17).

## 5. Packaging, install, and update

The extension is the plugin-equivalent bundling unit. Its `gemini-extension.json` manifest ([docs/extensions/reference.md](https://github.com/google-gemini/gemini-cli/blob/main/docs/extensions/reference.md), fetched 2026-07-17) carries `name`, `version`, `description`, `mcpServers`, `contextFileName` (extension-scoped GEMINI.md), `excludeTools` (down to command granularity, e.g. `run_shell_command(rm -rf)`), `settings` (user-configurable values with keychain storage for sensitive ones), `themes`, and `migratedTo`. One extension can bundle, side by side: a context file, TOML commands (`commands/`), MCP servers, skills (`skills/`), hooks (`hooks/hooks.json`), subagents (`agents/`), and policy rules (`policies/`). Variable substitution in manifest and hooks: `${extensionPath}`, `${workspacePath}`, `${/}`.

Install channel is Git-native: `gemini extensions install <git-url>` (also local paths), with `--ref` for pinning and `--auto-update`; `update <name>` or `update --all` refreshes; `link` symlinks a local checkout for development; `new` scaffolds from a template; `validate` checks structure; `enable`/`disable` toggle ([docs/cli/cli-reference.md](https://github.com/google-gemini/gemini-cli/blob/main/docs/cli/cli-reference.md), fetched 2026-07-17). Discovery runs through the extension gallery at [geminicli.com/extensions/browse/](https://geminicli.com/extensions/browse/) ([docs/extensions/index.md](https://github.com/google-gemini/gemini-cli/blob/main/docs/extensions/index.md), fetched 2026-07-17) - a browsable index, not a gatekeeping registry; the install source stays a Git URL. Plain-file installs also work at every layer: skills, commands, agents, and context files are all discovered from `~/.gemini/` and `.gemini/` directories without any extension wrapper (sections 1-2).

## 6. What is impossible

- **Per-command model override.** The TOML command schema has no `model` field; a command runs on the session's current model ([docs/cli/custom-commands.md](https://github.com/google-gemini/gemini-cli/blob/main/docs/cli/custom-commands.md)). Model choice per task is only available via subagents.
- **Subagent nesting.** "Subagents cannot call other subagents" ([docs/core/subagents.md](https://github.com/google-gemini/gemini-cli/blob/main/docs/core/subagents.md)) - an orchestrator subagent that itself dispatches workers is impossible; only the main agent fans out.
- **Parallel subagent waves.** Subagent execution is documented as single-threaded per instance ([docs/core/subagents.md](https://github.com/google-gemini/gemini-cli/blob/main/docs/core/subagents.md)); nothing documents a parallel dispatch primitive like Claude Code's multi-Task batch, so a review wave runs sequentially until proven otherwise.
- **Blocking at session end proper.** `SessionEnd` and `SessionStart` are advisory-only, no `decision` support ([docs/hooks/reference.md](https://github.com/google-gemini/gemini-cli/blob/main/docs/hooks/reference.md)). The gate lives at turn granularity (`AfterAgent`), which is what forge actually needs, but a hook cannot stop the user from quitting the CLI.
- **Stability guarantee for subagents.** The whole agents surface is behind `experimental.enableAgents` ([docs/core/subagents.md](https://github.com/google-gemini/gemini-cli/blob/main/docs/core/subagents.md)); an adapter that depends on it inherits Google's freedom to change the format.

## 7. Implications for a forge adapter

- **Spawn: available, adapter-required caveats are small.** Isolated-context subagents with per-agent `model` frontmatter cover forge's dispatch model; the no-nesting rule means forge's orchestrator must stay the main agent, which is already its shape. Sequential-only waves change crossfire's latency, not its semantics.
- **Enforcement tier: full.** `AfterAgent` blocking with automatic retry is a true Stop-gate; `BeforeTool` deny covers guard hooks; `BeforeAgent` `additionalContext` covers context injection. Gemini CLI lands in the same enforcement tier as Claude Code - hooks-backed, not prose-only.
- **Distribution: one extension repo.** Ship forge as a Git-installable extension bundling skills + hooks + agents in one manifest, with `--auto-update` as the update channel and the gallery listing as discovery. Skills themselves need near-zero translation thanks to the shared Agent Skills standard; the porting work concentrates in the hook protocol (JSON-over-stdio field names differ from Claude Code's) and rewriting agent frontmatter to Gemini's field set.
