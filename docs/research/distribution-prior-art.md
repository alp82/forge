# Cross-harness skill distribution prior art

Date: 2026-07-17. Question: how do existing projects ship agent skills/workflows across multiple harnesses - and which install/update mechanisms keep installed skills current versus letting them go stale? Wayfinder ticket #35. Feeds the install-and-distribution decision: "native channel per harness" vs "one universal npx-style installer" vs a hybrid.

Primary sources only: repo READMEs and source files, official harness docs, npm registry JSON, and the local machine's installed state (`~/.claude/skills/`, `~/.agents/`, `~/.claude/plugins/`) - all fetched or read 2026-07-17 unless dated otherwise. Sibling docs `matt-skills-patterns.md` and `landscape-survey.md` are cross-referenced, not re-derived. Every claim cites a URL or file path; version/date noted per source. House style: hyphens only.

## Executive summary

The field has converged on exactly two working distribution shapes, and the leaders ship both at once. Shape one is the **native channel**: every major harness now has a plugin/extension system with versioned install and managed updates - Claude Code plugin marketplaces ([code.claude.com/docs/en/discover-plugins](https://code.claude.com/docs/en/discover-plugins)), Gemini CLI extensions ([gemini-cli docs](https://github.com/google-gemini/gemini-cli/blob/main/docs/extensions/index.md)), Codex plugins ([developers.openai.com/codex/plugins](https://developers.openai.com/codex/plugins)), Cursor's marketplace ([cursor.com/docs/plugins](https://cursor.com/docs/plugins)); only opencode still has no registry, loading plugins from npm and skills from the filesystem ([opencode.ai/docs/plugins](https://opencode.ai/docs/plugins)). Shape two is the **universal installer**: the skills.sh CLI (`npx skills`, [vercel-labs/skills](https://github.com/vercel-labs/skills), v1.5.19 of 2026-07-16) copies skill folders into one canonical `.agents/skills/` dir and symlinks each detected harness's skill dir into it, tracked by a lockfile of source-repo tree hashes that powers `npx skills update`. Matt Pocock ships his 21 skills through both channels from one repo and states the tradeoff plainly: skills.sh copies you can customize vs a plugin that is "read-only bundles that auto-update" ([mattpocock/skills README](https://github.com/mattpocock/skills)). Superpowers answers cross-harness differently - not one installer, but one repo installable through ten harnesses' own native mechanisms ([obra/superpowers README](https://github.com/obra/superpowers), v6.1.1, 2026-07-02). The copy-vs-symlink verdict: native channels keep consumers current automatically (version-keyed cache, auto-update); the universal installer keeps *harnesses* consistent automatically (symlinks to one canonical copy) but keeps *upstream* current only when the user runs `update`; bare copy-in approaches (archived agent-rules, openskills without sync) go stale by construction. For forge - whose repo layout (`skills/*/SKILL.md` plus `.claude-plugin/`) already matches both channels' discovery conventions - the Pocock hybrid is the demonstrated low-cost path, with one caveat this survey measured locally: hooks ship only through native plugin channels, so any skills.sh route strips forge's enforcement layer.

## 1. The universal installer: skills.sh / `npx skills` (vercel-labs/skills)

The `add-skill` npm package is deprecated ("DEPRECATED: Use 'npx skills add' instead", v2.0.0, [registry.npmjs.org/add-skill/latest](https://registry.npmjs.org/add-skill/latest)); the live tool is the `skills` package, v1.5.19 (2026-07-16, 38 releases), repo [vercel-labs/skills](https://github.com/vercel-labs/skills), MIT, sole dependency `yaml`, bins `skills` and `add-skill` both pointing at `bin/cli.mjs` ([registry.npmjs.org/skills/latest](https://registry.npmjs.org/skills/latest)). The registry face is [skills.sh](https://skills.sh) - "The Agent Skills Directory", a leaderboard ranking skills by telemetry-reported install counts (top skill `find-skills` at ~2.5 M installs as of 2026-07-17; opt-out `DISABLE_TELEMETRY=1`).

**Install mechanics** (from `src/installer.ts`, `src/agents.ts`, `src/detect-agent.ts` in the repo):

- `npx skills add <owner>/<repo>` finds skill folders (dirs holding a `SKILL.md` with `name`/`description` frontmatter, searched under `skills/`, `skills/.curated/`, and agent-specific paths), **copies** them into a canonical dir - project `.agents/skills/<name>` or global `~/.agents/skills/<name>` with `-g` - then **symlinks** each detected harness's skill dir into the canonical copy (relative symlinks on Unix, junctions on Windows, copy fallback where linking fails). `--copy` mode skips the canonical dir and copies straight into agent dirs; the README recommends symlink as "single source of truth, easy updates".
- **Harness detection** is directory existence: e.g. Cursor if `~/.cursor` exists, Claude Code if the Claude home exists (respects `CLAUDE_CONFIG_DIR`), Codex respects `CODEX_HOME`; all checks run in parallel (`detectInstalledAgents()`). "Universal" agents that read `.agents/skills` natively (Cursor, Codex, opencode, Amp, Copilot, Cline) need no symlink at all; non-universal ones get links (Claude Code `~/.claude/skills`, Gemini `~/.gemini/skills`, Windsurf, Goose, Factory Droid, etc.). README claims 71+ supported agents (the count is the repo's own claim, not independently enumerated).
- **Lockfile**: `.skill-lock.json` (schema v3) at `$XDG_STATE_HOME/skills/` falling back to `~/.agents/.skill-lock.json`, per skill: `source`, `sourceType` (github/mintlify/huggingface/local), `sourceUrl`, `skillPath`, `skillFolderHash` (GitHub tree SHA), `installedAt`/`updatedAt` (`src/skill-lock.ts`).
- **Updates**: `npx skills update` compares the stored folder hash against the GitHub tree API without recloning, re-runs `add` for changed skills, and prompts removal for skills deleted upstream (`src/update.ts`). Update is **pull-only and user-initiated** - nothing runs it automatically.

**Local verification** (this machine, installed 2026-07-16): `~/.claude/skills/<name>` are relative symlinks into `~/.agents/skills/<name>` copies, and `~/.agents/.skill-lock.json` carries exactly the v3 schema above with `source: "mattpocock/skills"` entries and per-skill tree hashes (`ls -la ~/.claude/skills/`, `cat ~/.agents/.skill-lock.json`).

## 2. mattpocock/skills: one repo, two channels

The reference dual-channel case (repo created 2026-02-03, last push 2026-07-17, ~175 K stars per [api.github.com/repos/mattpocock/skills](https://api.github.com/repos/mattpocock/skills); plugin version 1.2.0 at main per [.claude-plugin/plugin.json](https://raw.githubusercontent.com/mattpocock/skills/main/.claude-plugin/plugin.json), last tagged release v1.1.0 of 2026-07-08). The README offers both installs side by side: `npx skills@latest add mattpocock/skills`, or `/plugin marketplace add mattpocock/skills` + `/plugin install mattpocock-skills@mattpocock`. The stated tradeoff: the skills.sh route yields editable copies, the plugin route "manages the skills as read-only bundles that auto-update" ([README](https://github.com/mattpocock/skills), also quoted in `docs/research/matt-skills-patterns.md` § 5). The repo pays for the hybrid with one layout: `skills/engineering/` + `skills/productivity/` (discoverable by the skills.sh crawler) plus a `.claude-plugin/` manifest, versioned via changesets (`.changeset/`, `CHANGELOG.md`). Payload is markdown-only - no hooks, so nothing is lost on the channel that cannot carry hooks.

## 3. Superpowers: one repo, ten native channels

[obra/superpowers](https://github.com/obra/superpowers) (v6.1.1, 2026-07-02, 14 skills) answers cross-harness without a universal installer: the README documents a per-harness install path for 10 harnesses, almost all through each harness's *own* plugin mechanism - Claude Code (`/plugin install superpowers@claude-plugins-official`, or via its own `obra/superpowers-marketplace`), Codex App (Plugins UI) and Codex CLI (`/plugins`), Cursor (`/add-plugin superpowers`), Antigravity (`agy plugin install <github-url>`), Factory Droid (`droid plugin marketplace add` + install), GitHub Copilot CLI (`copilot plugin marketplace add` + install), Kimi Code (`/plugins install <github-url>`), Pi (`pi install git:github.com/obra/superpowers`), and opencode via a bootstrap prompt ("Fetch and follow instructions from" the in-repo [.opencode/INSTALL.md](https://github.com/obra/superpowers/blob/main/.opencode/INSTALL.md)). On updates the README says only that they are "somewhat coding-agent dependent, but are often automatic" - i.e. it delegates the update story wholesale to each harness. The notable fact: the Claude-plugin format has become fetchable by other vendors' marketplaces (Codex, Cursor, Copilot, Kimi all install the same repo); how each non-Claude harness consumes the manifest was not verifiable from the repo itself. The opencode row shows the floor: where no channel exists, the install story degrades to "paste this prompt".

## 4. Other verified cross-harness distributors

- **openskills** ([numman-ali/openskills](https://github.com/numman-ali/openskills), npm v1.5.0, 2026-01-17, Apache-2.0, "Not affiliated with Anthropic"): `npx openskills install <source>` **copies** skills into `./.claude/skills/`, `./.agent/skills/` (`--universal`), or `~/.claude/skills/` (`--global`); `npx openskills sync` then injects an `<available_skills>` XML block into **AGENTS.md**, so any AGENTS.md-reading harness (Claude Code, Cursor, Windsurf, Aider, Codex named) can load skills via an `openskills read <skill>` CLI call - progressive disclosure re-implemented on top of the instructions file. Updates: `npx openskills update` re-pulls git-sourced skills; pre-source-tracking installs need reinstall.
- **rulesync** ([dyoshikawa/rulesync](https://github.com/dyoshikawa/rulesync), v14.0.1, 2026-07-17): the **generate** strategy - author maintains `.rulesync/` sources; `rulesync generate` emits per-tool files (CLAUDE.md, AGENTS.md, `.cursorrules`, `.github/copilot-instructions.md`, GEMINI.md, plus ignore files, MCP config, commands, subagents, skills, hooks) for a claimed 40+ tools. No runtime link: updates propagate only by re-running generate, but the output is native to each harness with zero adapter at load time.
- **steipete/agent-rules** ([repo](https://github.com/steipete/agent-rules), archived read-only 2026-05-03): the cautionary tale - shell-script copy-in of rules for Claude Code/Cursor, now frozen ("This was old stuff I used mid 2025"). Its successor [steipete/agent-scripts](https://github.com/steipete/agent-scripts) is **symlink-based**: a canonical repo linked into harness dirs by an idempotent `scripts/sync-skills`, with a documented harness asymmetry (Claude Code wants flat per-skill links under `~/.claude/skills/`; "Codex scans nested dirs, so it gets whole-root links"). Updates are `git pull` in the canonical clone - symlinks make every harness current at once.
- **AGENTS.md** ([agents.md](https://agents.md), repo [agentsmd/agents.md](https://github.com/agentsmd/agents.md)): the instructions-file convention underpinning several of the above - "a simple, open format for guiding coding agents", emerged from OpenAI Codex, Amp, Google Jules, Cursor, and Factory, now "stewarded by the Agentic AI Foundation under the Linux Foundation", with 20+ harnesses listed as supporting it. Claude Code is notably absent from the supporters list as fetched 2026-07-17. It distributes nothing itself but is the one file a cross-harness distributor can target once and reach many harnesses.

## 5. Native channels, one paragraph each

**Claude Code plugins** ([discover-plugins](https://code.claude.com/docs/en/discover-plugins), [plugins-reference](https://code.claude.com/docs/en/plugins-reference), fetched 2026-07-17). Install: `/plugin marketplace add owner/repo` (git URLs, `#ref` pinning, local paths) then `/plugin install name@marketplace` at user/project/local scope; `claude plugin install --scope project` non-interactively; projects can pre-declare `extraKnownMarketplaces` + `enabledPlugins` in `.claude/settings.json` so teammates get prompted on trust. Payload is the richest surveyed: skills, commands, agents, **hooks**, MCP servers, LSP servers, monitors, `bin/` executables, settings defaults. On disk: plugins are **copied** to `~/.claude/plugins/cache/<marketplace>/<plugin>/<version>/` - one directory per version, old versions orphaned then deleted after ~7 days (verified locally: `installed_plugins.json` v2 records `installPath .../forge/2.0.1`, `version`, `gitCommitSha`, `installedAt`). Updates: per-marketplace auto-update after session start (on by default for Anthropic-run marketplaces, **off by default for third-party**, toggleable, org-forceable via `autoUpdate: true`); manual `/plugin update`; the update key is `plugin.json` `version` (fallback: marketplace entry version, then commit SHA - an unset version makes every commit an update).

**opencode** ([docs/plugins](https://opencode.ai/docs/plugins), [docs/skills](https://opencode.ai/docs/skills), [docs/rules](https://opencode.ai/docs/rules)). Plugins are JS/TS files or npm packages named in `opencode.json` (`"plugin": ["opencode-wakatime"]`) or dropped into `.opencode/plugins/` / `~/.config/opencode/plugins/`; npm plugins are installed automatically with Bun at startup and cached under `~/.cache/opencode/node_modules/`. No central plugin registry is documented, and no pin/update command was found (unverified beyond the docs pages). Skills need no plugin at all: opencode discovers `SKILL.md` dirs from `.opencode/skills/`, `~/.config/opencode/skills/`, the Claude-compatible `.claude/skills/` and `~/.claude/skills/`, and `.agents/skills/` - meaning both the skills.sh layout and a Claude-style layout are readable natively. Instructions come from AGENTS.md (project, walking up) with CLAUDE.md fallback plus an `instructions` config array accepting globs and remote URLs.

**Gemini CLI extensions** ([extensions/index.md](https://github.com/google-gemini/gemini-cli/blob/main/docs/extensions/index.md) and [reference.md](https://github.com/google-gemini/gemini-cli/blob/main/docs/extensions/reference.md)). Install: `gemini extensions install <github-url|path>` (requires git; supports `--ref`, `--auto-update`, `--pre-release`); the CLI **copies** the extension to `~/.gemini/extensions/<name>/` with a `gemini-extension.json` manifest (`name`, semver `version`, `mcpServers`, `contextFileName` defaulting to GEMINI.md, `excludeTools`, keychain-backed `settings`). Extensions can ship context files, MCP servers, custom commands, themes, hooks, sub-agents, and agent skills. Updates: `gemini extensions update <name>` / `--all`, or per-extension auto-update if installed with `--auto-update`; the manifest `version` drives update detection; `gemini extensions link` gives developers a symlink install. Official gallery at [geminicli.com/extensions](https://geminicli.com/extensions).

**Codex CLI and Cursor** (for completeness - both grew native channels). Codex scans skills from `.agents/skills` up the tree, `~/.agents/skills`, and `/etc/codex/skills`, with a built-in `$skill-installer` pulling from the curated [openai/skills](https://github.com/openai/skills) catalog, and has **plugins** (skills + MCP + hooks + apps) cached at `~/.codex/plugins/cache/$MARKETPLACE/$PLUGIN/$VERSION/`, refreshed via `codex plugin marketplace upgrade`, listed in an official directory ([developers.openai.com/codex/skills](https://developers.openai.com/codex/skills), [/codex/plugins](https://developers.openai.com/codex/plugins)). Cursor runs a marketplace ([cursor.com/marketplace](https://cursor.com/marketplace), [docs/plugins](https://cursor.com/docs/plugins)) whose plugins bundle rules, skills, agents, commands, hooks, and MCP servers (`.cursor-plugin/plugin.json`), installed via marketplace UI or `/add-plugin`, with team marketplaces auto-refreshing every <=10 min. Neither is AGENTS.md-only anymore.

## 6. Copy vs symlink: the update problem

Four mechanisms observed, ordered by how automatically installs stay current:

1. **Managed copy with version-keyed auto-update** (Claude Code plugins, Gemini `--auto-update`, Cursor team marketplaces, Codex marketplace upgrade). The harness owns the cache, compares versions, and refreshes. Stays current without user action *where auto-update is on* - and for Claude Code third-party marketplaces it is off by default, downgrading this to "current after a manual `/plugin update`" ([discover-plugins](https://code.claude.com/docs/en/discover-plugins)).
2. **Symlink into a canonical store** (skills.sh default mode; agent-scripts). All harnesses are *mutually* consistent by construction - edit or pull the canonical copy once, every harness sees it. But the canonical store itself updates only on an explicit action: `npx skills update` (hash-compare against upstream) or `git pull` in the canonical clone. Symlinks solve fan-out staleness, not upstream staleness.
3. **Generate per harness** (rulesync). Native output, zero runtime dependency, but every upstream change needs a regenerate + commit in every consuming repo; staleness is invisible until someone reruns the tool.
4. **Bare copy-in** (openskills install, archived agent-rules, skills.sh `--copy` mode). Current at install time, stale forever after unless an update command re-pulls; the archived agent-rules repo is the end state.

Two locally measured wrinkles sharpen this taxonomy for forge:

- **Symlinks into a versioned cache are copies in disguise.** forge's own `/setup-forge` symlinks bare names into the installed plugin ("A symlink is always current - plugin updates propagate with zero re-run", `skills/setup/SKILL.md`), but the observed link target embeds the version: `~/.claude/skills/forge -> ~/.claude/plugins/cache/alperortac/forge/2.0.1/skills/forge` (`ls -la ~/.claude/skills/`). Claude Code caches each version in a separate directory and deletes orphans after ~7 days ([plugins-reference](https://code.claude.com/docs/en/plugins-reference)), so on version bump the link dangles until setup re-runs - the symlink-permanence claim in the setup skill and the README ("Plugin updates propagate on their own - no re-run needed", `README.md` § Install) does not hold across bumps under the versioned-cache scheme. The stale-copy nag in `hooks/session-start.sh` watches `.forge-version` stamps on copies but not dangling or version-pinned symlinks.
- **The two ecosystems already interoperate on this machine.** skills.sh-installed Pocock skills (symlinks into `~/.agents/skills/`) and plugin-installed forge skills (symlinks into the plugin cache) coexist in the same `~/.claude/skills/` directory - the flat-dir-of-symlinks pattern is the de facto merge point, and opencode reads both layouts natively ([docs/skills](https://opencode.ai/docs/skills)).

## 7. Comparison for the decision

| | Native channel per harness | Universal installer (skills.sh-style) | Hybrid (Pocock/superpowers shape) |
|---|---|---|---|
| **Install UX** | 2-3 commands, inside the harness (`/plugin marketplace add` + `/plugin install`; `gemini extensions install <url>` is one) | One command from any shell: `npx skills add alp82/forge`, auto-detects every present harness | Consumer picks: plugin commands or one npx line |
| **Update story** | Managed: version-keyed cache, auto-update (default-on only for first-party marketplaces; else one manual command) | User-initiated `npx skills update` (hash-diff vs upstream); symlinks fan the refresh to all harnesses at once | Plugin users auto-update; skills.sh users update on demand; two channels can skew versions |
| **Harness coverage** | One integration per harness; Claude Code today, each additional harness is a new manifest + release process (opencode has no channel at all) | Claimed 71+ agents from one repo layout; coverage is the installer's problem, not the author's | Full: plugin where channels exist, installer everywhere else |
| **Author maintenance** | Per-channel manifests, versions, and listings to keep in sync (forge already triple-mirrors its version per `CLAUDE.md` § Versioning) | Near zero: keep `skills/*/SKILL.md` discoverable; no manifest, no listing step; registry indexes on install | Both, but the payload is one tree; Pocock's whole overhead is one `.claude-plugin/` dir + changesets |
| **Failure modes** | Third-party auto-update off by default (silent staleness); versioned-cache symlinks dangle on bump (§ 6); channel lock-in per harness | **Markdown-only: hooks, agents, MCP config do not ship** - forge would lose its stop-gates and `hooks/session-start.sh`; updates never run unattended; telemetry-ranked registry, no review step | Version skew between channels; users installing via skills.sh silently get the hookless subset unless the README warns them |

The structural fact the decision should weigh: forge's payload is not markdown-only. `hooks/` (verify-tests, verify-build, block-git-writes, session-start) ships only through the plugin channel - Claude Code plugins, Codex plugins, Cursor plugins, and Gemini extensions all carry hooks ([plugins](https://code.claude.com/docs/en/plugins), [developers.openai.com/codex/plugins](https://developers.openai.com/codex/plugins), [cursor.com/docs/plugins](https://cursor.com/docs/plugins), [gemini-cli reference.md](https://github.com/google-gemini/gemini-cli/blob/main/docs/extensions/reference.md)); the skills.sh pipeline carries only `SKILL.md` folders ([vercel-labs/skills](https://github.com/vercel-labs/skills) `src/installer.ts`). A universal-installer-only story therefore ships forge minus its enforcement layer - the exact layer `docs/research/landscape-survey.md` § 2 identified as forge's rarest asset. The demonstrated end state among the adoption leaders is the hybrid: one repo whose layout satisfies both channels, plugin-first where hooks matter, skills.sh for reach.
