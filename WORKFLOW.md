# Global Development Rules

## Principles
- Never guess, never assume, never improvise unagreed solutions.
- Name the unknowns and say when you don't know. "I couldn't confirm X" is a complete answer; silent confidence on an unverified point is the failure to avoid.
- Extracting actual intent is more important than moving fast.
- Research before asking. Subagents exhaust filesystem, tools, and web first; questions only surface what those sources don't already answer.
- Clarify in loops, not single passes. Intent and clarification steps re-run with prior rounds folded in until the latest exchange surfaces no new aspects. Loops within one step are free and do not count as backward edges - a convergence loop, distinct from a correction revision (`## Revision Contract`).
- Leave touched code better than you found it. Unrelated changes get their own task.
- No TODOs, placeholders, or incomplete implementations. No backwards compatibility - obsolete code gets deleted, not preserved. No unnecessary comments, docstrings, or type annotations on unchanged code.
- Always use the editor's dedicated file operation tools. When an edit fails, fix the edit - never fall back to shell commands (sed, awk, python scripts) for file manipulation.
- Prefer the Grep and Glob tools whenever they are granted - they never shell-parse the path or pattern. Fall back to `rg`/`grep`/`find` via Bash only when those tools are unavailable or you genuinely need a pipe; quote globs (`rg -g "*.md"`) so the same command works in bash, zsh, or fish.

## Tone
- No corporate phrasing or fake contrast framing ("While X is important, Y...").
- No sycophancy. If an idea is weak, say so and explain why.
- Wit and sarcasm welcome when they land. Don't force it.
- Be direct. Say what you mean.

## Formatting
- No preamble or postamble. Start with the answer, end when the answer ends. Don't restate the user's question before answering.
- Status updates between tool calls: a status update that reports a state transition speaks the status line - `<state> ▶ <next action>` in plain words, or the note form `<event> (<reason>)` - grammar per `doctrine/render-card.md`; a status update with no transition to report (a talk-path turn, a mid-work aside) stays a terse plain fragment. Full sentences are reserved for points where the user must decide or act; self-narration is prohibited. The pipeline loop is card-only (`### The loop`).
- Turn zero - the moment a pipeline request lands, before the router is called - has exactly one surface: the status line `request received ▶ triage`, then the first render card. Everything before the card (reading doctrine, opening files) is silent. This closes the gap the loop's card-only rule leaves open, since that rule is scoped to steps 1-4 and turn zero precedes them.
- Cut hedges and qualifiers that don't change meaning ("I think maybe we could possibly" → "we can"). Keep hedges only when the uncertainty is real and load-bearing.
- Exceptions: code, commits, PRs, docs, and high-stakes confirmations (destructive ops, security warnings, irreversible actions) follow their own conventions - use full prose when clarity matters more than brevity.

## Context Discipline
- Always prefer subagents. The main agent orchestrates and talks to the user - it does NOT read entire codebases, do deep analysis, or implement large changes itself.
- Subagents return structured verdicts (VERDICT/FINDINGS/ACTION_NEEDED), not raw dumps. A return is the agent's *conclusion* - the wrapped output block its definition specifies, nothing more: no narration of reads, no restating inputs. The orchestrator relays only what the next stage or the user needs. A verbose return is a defect - it burns the main context the orchestrator exists to protect.
- Spawn dynamic subagents when the situation calls for it. Pick the cheapest model that can handle the job.

## Subagent Context Inheritance

Subagents inherit nothing automatically - MEMORY.md, its linked files, and project-level docs do not transfer. The plugin's **PreToolUse(Agent) hook** (`user-context-injector`) prepends up to two independent blocks to the Agent prompt: `## USER_CONTEXT` (MEMORY.md + linked files, durable user preferences and feedback; **user-aware** means the agent receives it, the hook's case statement is the allowlist, and a user preference beats a default behavior unless it creates a correctness issue) and `## PROJECT_CONTEXT` (matching slices of the project's `docs/` folder; **project-aware** means the agent receives it, and the hook's `READ_MAP` lists which doc tokens each agent needs: `intent`, `stack`, `glossary`, `adrs`). `hooks/user-context-injector.sh` is the single source of truth for both allowlists (agent files carry no `reads:` field); missing sources skip silently - no errors, no scaffolding prompts.

### Project Context docs

Project intent, stack choices, glossary, and prior architectural decisions live in your repo's `docs/` folder. Four tokens feed it: `intent` -> `docs/INTENT.md`, `stack` -> `docs/STACK.md`, `glossary` -> `docs/GLOSSARY.md` (each injected as its full body), and `adrs` -> `docs/adr/*.md` (a summary list - one bullet per ADR with status, title, summary, path - to keep prompts lean). The hook drops ADRs with status `deprecated` or `superseded`, files matching `0000-*.md`, and ADRs whose summary still contains a `_TODO:_` marker. Templates ship in the plugin's `templates/` folder; run `/alp-river:setup` to populate INTENT/STACK/GLOSSARY interactively, and `/alp-river:adr` to record a decision deliberately - it drafts via the `adr-drafter` agent (read-only, opus) and rejects duplicates of active ADRs.

## Confidence Tagging

See `doctrine/confidence-tagging.md` - injected into citing agents by the PreToolUse(Agent) hook.

## Pipeline

Every code-modifying request runs through a **composed route** - **no exception for prompts that feel small**; a trivial ask just produces a tiny route (often one stage). The deterministic router (`hooks/route.py`) assembles the exact stages a task needs from the catalog (`generated/catalog.json`, compiled from `agents/*.md` frontmatter - see `doctrine/CATALOG.md` and `doctrine/SIGNALS.md`) and grows or shrinks it as the task reveals itself. There is no per-tier step list; size (XS-XXL) is a *readout* of the assembled route, not a driver.

### The loop

The main agent is a thin orchestrator. It holds four pieces of run state - `live` (signal topics currently emitted, seeded with `request-received`), `available` (artifacts that exist, seeded with `request`), `ran` (stages already executed), and `premises` (the assumptions the current route was built on; each stage is handed these and reports breaks) - and turns the crank.

Each turn (the loop is **card-only**: the render card carries the why, mechanism reasoning stays in the thinking block, and the orchestrator narrates none of the route/render/read-contract/spawn/recompose mechanics inline - the reserved inline-prose bucket is actionable-only: findings, errors, stalls, and scope-shifts the loop must not swallow surface in status-line grammar per `doctrine/render-card.md` (note form when non-actionable), full sentences only where the user must decide or act, and `triage`'s one-line restatement and gate questions keep their own forms):

1. **Route.** Silently call the router **once** on the current state; it returns the ordered route + size + `triggered_by`. This single call is also the recompose - step 4 updated the state at the end of the previous turn, so a freshly published signal grows the route here, a `scope-shift` reshapes it, and sticky stages persist (never auto-dropped). One router call per turn, not two. JSON in, JSON out:
   ```
   echo '{"live":[...],"available":[...],"already_run":[...]}' | python3 hooks/route.py
   ```
   The router rejects an unrecognized top-level key: it writes the offending key to stderr and exits nonzero rather than returning an empty route, so a malformed call is never mistaken for convergence.
2. **Render.** Show the route as **native markdown** - emitted directly from state, rendered (not fenced), no script, no Bash. The render card IS the narration for the route/recompose. Render **every turn** so progress is never invisible: the full route on the first turn and at any gate (legibility A), the delta on a plain recompose (legibility B). This is a status surface, not a question - asking is what a gate stage does when it runs (step 3, see `## Gates`). Formats:
   - **Full (A)** - a header `path · size · N stages`, then the route as a phase-grouped nested list - grammar, markers, and phase order per `doctrine/render-card.md`. The router returns `waves` (the topo levels); stages in one wave share no precedence edge and dispatch as a single parallel batch, each marked `▶`. The render composes `route` + `held` so a gated stage stays visible. At a plan-approval gate the card also carries the producer's Plan Breakdown as a short **Plan breakdown** section beneath the list. Example, emitted exactly as this native markdown (no fence):

     *code · S · 3 stages*

     - 🔎 **Intent**
       - ✓ triage
     - 🔨 **Build**
       - 🔒 code-implementer (waiting on an approved plan)
     - 🔬 **Review**
       - • correctness-reviewer

   - **Milestone layer** - during a milestone-loop build the card adds the milestone-status layer (the `milestone k of N` header and the `🟩 🟨 🟥` list) atop the per-stage markers; shape per `doctrine/render-card.md`, arming per `## Milestone loop`.
   - **Delta (B)** - lead with the plain-word state (the new signal's meaning in words), then the route change: `+added / -removed (now size/N)`, as one native-markdown line - example: `auth surface touched ▶ +security-reviewer (now L/6)`. Grammar per `doctrine/render-card.md`.
3. **Run** the next not-yet-run stage in route order. Held stages are NOT in `route` - a lock keeps a held stage out of the dispatch list until its `until` signal fires (see `## Locks`). Silently spawn the stage's agent at the cheapest model its frontmatter allows and silently read its input contract; hand it the artifacts its `input` names plus the `premises` - each input the **verbatim output** of the stage that produced it, never paraphrased, stubbed, or hand-written (Input Template Contract). If an `input` does not exist yet, run its producer - **never fabricate a predecessor's artifact** to start a dependent stage early. When several stages produce the same artifact (the three domain prototypers all emit `@prototypes`), concatenate each firing producer's verbatim output into the one slot - never pick one and drop the rest. Independent stages that share inputs (the Review wave over `@diff`) may run as one parallel batch; a stage whose `input` is another stage's `output` waits for that producer - the `input`/`output` precedence graph is the one rule never bent (`doctrine/CATALOG.md`). Expect a **structured return, not a transcript** (Context Discipline); surface only the line(s) the render, the next stage, or a gate actually needs. **Hang-prone stages run in the background**: the network/tool-budget stages (researcher, investigators, web-using reviewers, every parallel review wave - one hung member otherwise freezes the batch) and the side-effecting stages (`code-implementer`, `fixer`, `test-author`, `system-executor`), whose working tree is exactly the durable record `## Recovery` reconciles from. Dispatch each with `Agent(run_in_background: true)`, keep its task handle, and keep cranking the loop while it runs - a backgrounded stage is dispatched-and-awaited across turns, not run-to-return inside one; this is the one case where a stage spans turns. Watch it two ways: the completion push is the normal end, and a watchdog catches the stage that never pushes - a transcript mtime freeze (no change for ~120s; file metadata only, never content) or an absolute wall-clock deadline, whichever fires first - then stop the task (`TaskStop(task_id)`) and treat it as a missing output per `## Recovery`: a read-only stage re-runs once then surfaces (the same stage re-run, not a backward edge), a side-effecting stage reconciles from the working tree it left behind. A live task handle lost to compaction is the same missing output (`## Compaction`). Where the runtime lacks background Agent dispatch, run the stage in the foreground - the degraded fallback: the agent's OWN tool budget is the sole cap, and the missing-output rule catches an empty or explicitly-partial RETURN. Cheap deterministic stages (triage, the router call) run in the foreground as before.
4. **Update.** Silently add the stage's `output` artifacts to `available`, its published `#signals` to `live` (each carries a one-line message = the why), and record it in `ran` - two different keys, never conflated: `triage`'s artifact `@confirmed-intent` lands in `available`, its signal `#intent-confirmed` in `live`. A published `scope-shift` logs a broken premise. A reviewer's published signal is read per `## Convergence`. These updates are the state the next turn's step 1 recomposes from. **Then persist the run state** - the orchestrator writes the run-state snapshot to disk with its own native Write tool, a HARD REQUIRED step-4 action. The target is the per-run path the SessionStart recovery hook injects into context: `<cwd>/.alp-river/runs/<run-id>/run-state.json`, where `<run-id>` is the session id - the orchestrator is told the path, it does not invent it. If the file already exists and this session has not yet touched it (a resumed or recovered run), Read it once first - a prior Read is what lets the Write tool overwrite an existing file - then Write the full overwrite: the latest snapshot replaces the prior one in place. The snapshot is exactly `{schema_version, run_id, cwd, route, live, available, ran, premises, mid_run_stage, pending_gate, pending_gate_question, artifact_index}`: `route`, `live`, `available`, and `ran` are JSON ARRAYS, `artifact_index` is an object, `schema_version` a number, the rest strings. There is no `written_at` field, freshness is the FILE's mtime, set by the OS on each write. Keep the free-text fields terse and quote-light: the orchestrator authors this JSON, and since the read path validates and silently falls back, a malformed write costs durability invisibly - terseness is the mitigation. The write is silent and the loop stays **card-only** - no inline `<RUN_STATE>` block is ever emitted. Durability rests on this write being a hard step-4 action AND on `already_run`/`live`/`available` accuracy - NOT on the age guard catching a missed write: the backstop is route-recomputation plus reconcile-from-tree. A failed Write is best-effort with no retry clause: the next turn's hard step-4 write supersedes it - except a Write rejected for overwrite eligibility, where the orchestrator Reads the file once so that superseding write can land.

Repeat until **convergence** - defined once in `## Convergence`.

### Seed and path

The route is rooted by the always-on `triage` stage. It reads the request and publishes exactly one **path** - `talk`, `sketch`, `code`, or `system` - plus early signals (`ambiguous`, `novel-domain`, a bug-framing `bug`, risk sniffs) and one advisory `est-size:<tier>`. The router composes from there. The path is sticky but re-evaluated every turn: `talk` flips to `code`/`system` on "do it"; a `sketch` graduates on the kept work. A path is defined by **what it leaves behind**:

- **`talk`** - inline-first; leaves nothing. The main agent answers directly and reads freely; the path is parked. Only the expensive moves ask first - spawning a recon subagent (`discuss`, `researcher`, an investigator, `design-prototyper`) or producing a diagram/visual - each a one-tap confirm. Nothing produces a `diff`, nothing is reviewed or documented.
- **`sketch`** - leaves a sandboxed throwaway (`.prototypes/`) in any medium: a code tracer-bullet (`sketch-build`), a diagram, a UI mockup, an idea sketch. Runs relaxed; the code-only ceremony band is filtered off the path by each stage's `routes`. Correctness and security still apply. Graduating flips to `code`/`system`.
- **`code`** - leaves a reviewed change in the codebase via the full composed route. A bug is a code build: `triage` pairs `code` with a `bug` signal and the `code-investigator` diagnoses inside the route - there is no separate `diagnose` path. Two independent axes shape a code build: `#needs-tests` (the TDD axis - a change carrying real logic) and `#significant-build` (the review-depth axis - a serious build that pulls Scout and the deep lenses). The `code-implementer` carries the TDD gate and the plan gate (`## Locks`).
- **`system`** - leaves a verified change to the machine: configs, troubleshooting, CLI tooling. `system-planner` (ordered steps, each with backup / dry-run / rollback) -> `system-executor` (runs them) -> `system-verifier` (confirms the desired state actually holds). A bug pairs `system` with `bug`: the `system-investigator` diagnoses from service state, logs, and configs. No TDD chain and no code-convention lenses; the executor carries the safety gate and the always-confirm plan gate (`## Locks`), so the system path carries no `#significant-build`. Security still applies.

### Worked routes

Six `echo STATE | python3 hooks/route.py` traces:

- **code** - `{"live":["code","needs-tests","significant-build","plan-ready","plan-approved","code-written","ui-touched","auth-surface","perf-surface"],"available":["confirmed-intent","diff"]}` composes Scout (`reuse-scanner` + `health-checker` + `prototype-identifier`, pulled by `#significant-build`), the full Review wave (correctness via `#code-written`; simplicity via `#plan-ready`; shape, conventions, acceptance via `#significant-build`; performance via `#perf-surface`; test-gap, test-verifier via `#needs-tests`; ux, accessibility, design-consistency via `#ui-touched`), `security-reviewer` via `auth-surface`, and `capture-agent`. The lenses share `@diff`, so they land in one parallel wave. Size XXL.
- **trivial code** - `{"live":["code","direct-impl"],"available":["confirmed-intent"]}` composes exactly `code-implementer` (size XS) - the short path skips the planner entirely. Both locks are inactive: no `#needs-tests`, no `#plan-ready` - the implementer runs straight off `@confirmed-intent`, unheld. Once `#code-written` makes `@diff` available, `correctness-reviewer` joins; `simplicity-reviewer` does NOT (it subscribes `#plan-ready`, never published here).
- **small planned code** - `{"live":["code","intent-confirmed"],"available":["request","triage-read","confirmed-intent"]}` composes only `code-planner` (size XS). Once the planner publishes `#plan-ready`, the next recompose returns `held: {"code-implementer": ["plan-approved"]}` (TDD lock inactive, plan gate holds). The orchestrator emits `#plan-approved` (auto on this small plan, or a one-tap confirm), releasing the implementer; `correctness-reviewer` and `simplicity-reviewer` join once `#code-written` makes `@diff` available. The deep lenses, Scout, clarify, test-chain, challenge, and Document wait on `#significant-build` (the TDD chain on `#needs-tests`).
- **sketch** - `{"live":["sketch","code-written"],"available":["confirmed-intent","diff"]}` composes just `sketch-build` then `correctness-reviewer`; the code-only lenses drop `off-path` via the `routes` filter. Size S.
- **system** - `{"live":["system","plan-ready","destructive-op"],"available":["confirmed-intent","system-plan"]}` composes `safety-gate` (armed by `destructive-op`) and holds `system-executor` on BOTH unmet untils - `held: {"system-executor": ["safety-approved", "plan-approved"]}`. The `safety-gate` clears `#safety-approved` in-route; the orchestrator emits `#plan-approved` via the one-tap pre-execution confirm. Both fired, the executor runs and `system-verifier` follows. Without `#destructive-op` the plan gate still holds the executor alone.
- **talk** - `{"live":["talk","ambiguous"],"available":["request","triage-read"]}` composes `clarifier` (pulled by `ambiguous`) then `discuss`, ordered after it because `discuss` optionally consumes the clarifier's `confirmed-intent`. No diff, nothing reviewed.

### Intent

`triage` settles framing. When the request is clear, **state the one-line interpretation and proceed** - no confirmation gate; the user corrects in their next message if it is wrong. `triage` mints `@confirmed-intent` on a clear code or system ask, so its Scout step is satisfied without the clarifier. When `triage` publishes `ambiguous` (any genuine doubt - low bar), the `clarifier` stage joins and loops until intent is settled. Intent is always *stated*, never silently assumed (see Principles), but a clear ask is not stopped.

### Gates

A gate is a stage whose output is a **user decision**, rendered via `AskUserQuestion` (Concise Surfacing Contract). It fires only when its triggering signal is live AND the answer could change what happens next - a 95%-Continue checkpoint is surfaced as a status line, not asked. The user's choice publishes a signal (`approved` / `scope-down` / `abandon` / ...) that feeds the next recompose. An `abandon` outcome is a **run-terminal**: the orchestrator stops the route, drops any stage still held behind the abandoned gate (its `until` signal will never fire), and surfaces what ran and what did not - a `safety-gate` Abort terminates cleanly rather than leaving the held `system-executor` waiting forever.

The **plan-approval gate** holds both implementers behind `#plan-approved` (see `## Locks`). On the `code` path with `#significant-build` live, `plan-challenger` runs in-route and its Approve publishes `#plan-approved`; on a multi-plan run (`doctrine/multi-plan.md`) the per-plan challengers run critique-only and `plan-arbiter` is the in-route publisher, on its Adopt verdict. On the system and small planned build paths there is no in-route publisher, so the orchestrator emits `#plan-approved` itself per the `## Locks` Release policy: a one-tap confirm before execution (system: always; small planned build: only when the plan touches `>1` file or `est-size > S`, else auto-published) - the De Morgan dual of the `## Locks` threshold (`<=1` file AND `est-size <= S`), kept in sync by this cross-reference; see that section for the absent-est-size default and the confirm wording. The `#direct-impl` short path has no plan gate at all.

### Asymmetric rigor

Skipping a stage needs a positive signal; adding one needs only doubt. Safety and clarify stages carry `guard: sticky` or fire on any risk sniff - in by default, dropped only on strong signal. A needless question costs mild annoyance; a wrong assumption costs the task.

## Model Tiering

Each stage declares its own `model` in agent frontmatter, across four tiers. `fable` (Mythos-class) is reserved for the deepest generative and adversarial reasoning: the code planner, the plan challenger, the plan arbiter, and the code implementer. `opus` carries the judgment stages - the intent loops, investigation, the deep review lenses, the design and UX explorations, the system planner, and the authoring stages. `sonnet` runs the mechanical executors, the scan stages, and the surface review lenses. `haiku` handles classification and lookups. The router spawns each stage at its declared model; there is no per-tier override table - to retier a stage, change its `model` in frontmatter.

Alongside `model`, each stage declares an `effort` level matched to the JOB, not to the model: model tracks how much unstructured judgment a job needs, while effort tracks whether the job generates, judges, executes, or classifies. The enumerated levels are `low`, `medium`, `high`, `xhigh`, and `max` - `xhigh` sits between `high` and `max`, and neither is currently declared by any stage: `xhigh` is held in reserve as the documented best setting for coding and agentic work, and `max` is avoided because it is prone to overthinking with diminishing returns. Mechanical stages that execute an upstream decision sit at `medium`; analysis, the review lenses, and the intent loops at `high`; the authoring stages (`setup-agent`, `adr-drafter`) at `medium`; and `capture-agent` at `low`. The four `fable` stages currently run at `medium` as a deliberate experiment: Fable at low-to-medium effort is documented to match or beat prior-generation models at their top effort, and shorter turns reduce the risk of the loop's transcript-freeze watchdog (`## Pipeline` > The loop, step 3) killing a backgrounded planner mid-think. Effort sets a stage's thinking budget independent of its model and is read by the harness at spawn, never by the router or catalog. It is model-gated: `haiku` does not honor effort, so the haiku classification stages (`triage`, `prototype-identifier`, `health-checker`) carry no `effort` line.

## Instruction-to-hook

Promote a repeated agent instruction to a deterministic hook when the check is mechanical, has a single correct answer, and recurs across runs; keep it an agent instruction when it needs judgment or run-specific context. Example: "the build and tests must be green at finish" is mechanical and recurs, so it lives in the `verify-build` / `verify-tests` Stop hooks rather than in each implementer's prompt.

## Clarification Loops

The single `clarifier` stage runs as a loop carrying both clarify altitudes (direction/intent-level and detail/requirements-level). It loops *internally* (the route sees one stage) until its exit criteria hold: `VERDICT: clear`, `NEW_ASPECTS_FOUND: no`, and the user has no further additions. **No round cap** - convergence governs it, not a budget; every round surfaces its open questions, so the loop never spins silently and the user can direct the agent to proceed at any point. Re-invocations carry `<PRIOR_ROUNDS>` (a compressed Q&A log) so settled points are never re-asked. Direction questions queue ahead of detail, so a scope answer can dissolve a detail question before a slot is spent. Before asking, the agent exhausts filesystem and web sources and reports `LOOKUPS_PERFORMED`; if research already answers a question, it drops it.

## Concise Surfacing Contract

- **Purpose + MUST-render rule**: inline prose stays only for decisions. When a gate stage is triggered (its signal is live) AND the answer could change the outcome, the orchestrator MUST invoke `AskUserQuestion` rather than render options as prose - the reasoning lives in `description`/`preview`, not inline; when no picker-eligible item is open and the stage's exit criteria hold, proceed without prompting. `triage`'s one-line intent restatement is plain text, not a picker.
- **Picker-eligible sources**: `clarifier` (`QUESTIONS`, open `DEFERRED_QUESTIONS`, promoted `[unsure]` criteria or assumptions); `design-prototyper` (`PARAMS_TO_CONFIRM`); `ux-prototyper` (`FLOW_TO_CONFIRM`); `plan-challenger` (`CHALLENGE_QUESTIONS`); and any gate stage whose output is a user decision.
- **Question schema, description vs. preview**: `question` (text), `header` (max 12 chars), `multiSelect`, `options` (2-4; each `label`, `description`, optional `preview`). `description` carries the essence plus, for any non-trivial decision, one concrete example of what the choice produces (e.g. `wrapped -> {users:[...]}` vs `bare -> [...]`) - the DEFAULT shape for every surfaced explanation at a decision point: plain words first, then a concrete before->after example, jargon cut. The example is load-bearing - lead with it. Bare yes/no and fixed-process gates (Continue/Stop, Approve/Revise/Reshape) are exempt: their concrete example lives in the card's Plan breakdown instead. `preview` is best-effort enrichment - never load-bearing. The CLI supplies the "Other" escape; agents MUST NOT synthesize their own.
- **Plan breakdown**: at every plan-approval gate where the user is asked, the orchestrator renders the producer's Plan Breakdown verbatim inside the Full (A) render card - native-markdown prose, authored plain by the producer with a concrete example and a small visual woven in; the orchestrator relays, never synthesizes. A silent auto-approve renders no breakdown.
- **4-question cap + DEFERRED priority queue**: `AskUserQuestion` takes 1-4 questions. With more eligible items, fill the slots in order - open `QUESTIONS`, then `[unsure]` criteria, then `[unsure]` assumptions - and roll the rest into `DEFERRED_QUESTIONS`, preserving order. It rides inside `<CLARIFY_OUTPUT>` so it survives compaction; still-open items thread into the next round's `<PRIOR_ROUNDS>`.
- **Challenger reshape**: `plan-challenger`'s `CHALLENGE_QUESTIONS` carry Approve (proceed) / Revise (rerun planner with `BLOCKERS`) / Reshape (back to intent). `SCOPE_MISMATCH` surfaces inline alongside `BLOCKERS` and in the Reshape option's `preview`.

## Convergence

A route is **done** when the router returns an empty route AND an empty `held` map and
every Review lens that ran came back `clean` - read from its explicit `SIGNALS_PUBLISHED:`
token (`doctrine/reviewer-contract.md` ### Published-signal line), never inferred from
VERDICT prose. A non-empty `held` map is never convergence - it is either a pending
plan-approval the orchestrator owes (emit/surface it, see Stall guard below) or, on
`#abandon`, a dropped stage.

- There is no edge budget. The only loop guard is oscillation - a `scope-shift` that re-fires without resolving is surfaced to the user, not retried silently. A correction revision (a challenger `revise`, an implementer kickback) re-spawns its producer per `## Revision Contract`. A *missing* output block is not convergence either: a stage that returns no wrapped output - including the harness `[Tool result missing due to internal error]` - is a failure, recovered per `## Recovery`, never an open-ended wait; a background deadline breach and a task handle lost to compaction collapse to the same case.
- **Stale-approval retraction.** On a **pre-implementation** re-plan (a challenger `revise`, a planner kickback, a reshape that re-derives the plan before the implementer or executor ran), the orchestrator performs a **live-set removal** of `#plan-approved` so the revised plan must re-earn it through its gate - the explicit retraction primitive, counterpart to dropping a stage from `already_run` (e.g. `system-verifier` after a drift fix). Once the implementer/executor has run, a late re-plan is a forward correction, not a re-gate. Inside a milestone loop this clause reads per-milestone (`## Milestone loop`).
- **Milestone HARD withhold.** During a milestone loop `@diff` and `#code-written` are withheld until `#milestones-complete` (`## Milestone loop`), so the End Review lenses are dropped as unsatisfiable per milestone - dropped, not deferred - and fire once at loop end.
- **Late escalation, never a re-gate.** A signal published late by `correctness-reviewer` on a cheap-path build escalates review depth without re-holding the already-run implementer: a late `#needs-tests` pulls the TDD chain in to retroactively test the diff, a late `#significant-build` pulls the deep lenses (plus the `fixer`). Neither re-gates through `plan-challenger` - the plan gate is strictly pre-implementation.
- **Multi-plan adjudication.** On an armed multi-plan run (`doctrine/multi-plan.md`) the orchestrator publishes `#critiques-ready` AND seeds `@competing-plans` + `@plan-critiques` into `available` in ONE atomic recompose, after the per-plan critique phase. The arbiter's Adopt publishes `#plan-approved`; Revise-first and Hybrid both re-spawn the planner via `## Revision Contract`, so the revised plan re-earns approval through its gate.
- **Stall guard.** A stage held on `#plan-approved` with no in-route producer AND no plan-approval confirm pending is a stall, not convergence - surface it to the user, never wait forever. On the system path the operative test is "no confirm pending": an owed, unsurfaced `#plan-approved` emit is the stall to catch. On a multi-plan run the arbiter IS an in-route producer, so its held state is a non-stall - but only if the atomic co-publish landed: `#critiques-ready` without the seeded artifacts drops the arbiter as unsatisfiable-input, nobody owes the emit, and that state IS a stall to surface.
- `system-verifier` works the loop in reverse: on drift it publishes `findings:system`, which re-runs the `system-executor`; the orchestrator then drops `system-verifier` from `already_run` so it re-verifies the corrected state - the system analog of a review lens re-running after the `fixer`.

## Recovery

When a stage's output is lost - a deadline breach (`## Pipeline` > The loop, step 3), a task handle dropped to compaction, or a harness `[Tool result missing due to internal error]` - the orchestrator reconstructs what happened from the durable record and continues on its own, never an open-ended wait. The branch is decided by what the lost stage *does*; the two branches are complementary and exhaustive (every stage is either read-only or side-effecting), and detection only has to *eventually* fire - reconcile-from-trace carries the correctness, the timer just decides when to look.

- **Re-runnable stage** (researcher, the investigators, every review lens - all read-only). Re-dispatch it: idempotent, so recovery is a single bounded retry; if that retry also breaches, surface the stall to the user.
- **Side-effecting stage** for code and tests (`code-implementer`, `fixer`, `test-author`). The working tree IS the durable record - reconcile from it rather than re-run blind: `git diff` the tree against the approved plan's expected file set, then run the project gate (build / type-check / the test suite). Four outcomes: **COMPLETE** (expected files present, gate green - publish the stage's normal output and proceed), **PARTIAL** (some present - resume the remainder; the idempotent read-modify-write each implementer/fixer leans toward, `doctrine/code-doctrine.md`, lets a re-dispatched remainder no-op work already on disk), **NOTHING-LANDED** (re-dispatch as if it had not run), **AMBIGUOUS** (the tree diverges in a way none of the three explain - surface to the user; the only branch that pulls the user in).
- **Side-effecting stage** for system (`system-executor`). Reconcile through the `system-verifier` reverse-loop (`## Convergence`): its drift finding re-runs the executor. No separate mechanism.

## Durability

A hard session kill - not just compaction - is recoverable because the orchestrator persists its canonical run state to disk every turn. The snapshot lives at the per-run path `.alp-river/runs/<run-id>/run-state.json` (`<run-id>` = session id), so two pipelines in one repo never clobber one file. The orchestrator writes the snapshot with its own Write tool as the HARD REQUIRED step-4 action (loop step 4 owns the write contract and the exact schema). The SessionStart recovery hook `hooks/recover-run-state.sh` (registered on startup, resume, and compact) reads it back and always injects the per-run write path so the orchestrator learns where to persist. A companion hook, `hooks/ensure-gitignore.sh` (startup, resume), adds `.alp-river/` to the repo's `.gitignore` when absent; if the directory was already committed it injects a turn-1 instruction to surface an untrack command for the user (git rm is user-only, per the git-write guard).

- **Read-back guards.** A candidate run-state.json is offered only when it passes every guard: **G1** validate-on-read - `jq`-parseable AND `schema_version==1` AND the required fields present; **G2** its `cwd` equals the session cwd; **G4** its file mtime is within `RIVER_STATE_MAX_AGE_SECONDS` (default 86400) of now - freshness is the file's OS mtime, there is no `written_at` field. Selection is per-source: resume reads ONLY its own `runs/<session_id>/` dir; startup SCANS `runs/*/` and picks the freshest passing candidate. A **converged** state (empty route AND empty/absent pending gate) is never offered, and dirs older than the max-age are **pruned** during the read pass. A passing candidate injects a recovery offer ("resume at <stage> on route <route>, or start fresh?") as a turn-1 instruction; SessionStart additionalContext injects SILENTLY (no picker can render), so the offer is best-effort - ignored or compacted away, it is a silent no-op. Resume assumes the harness preserves the session_id across a hard-kill -> resume cycle; a newly minted id falls back to the next genuine startup scan - safe degradation.
- **Honest scoping.** A missed write leaves a recent-but-stale file G4 would pass - the durability contract and its real backstop live in loop step 4. The live background-task handle stays non-recoverable - reconcile the mid-run stage from the working tree. A malformed write degrades to anchor-only with the preserve-manually note. A pipeline started right after a mid-session `/clear` is non-durable until the next genuine session start - the recovery hook deliberately skips `clear`. Only a pending plan-approval gate is re-emitted (the hook carries its plan handle path from the artifact index); any other mid-flight picker simply re-runs its read-only stage.

## Locks

A stage may declare a scheduling gate in frontmatter:

```yaml
lock:
  - while: '#needs-tests'
    until: '#tests-ready'
```

Three states: **Inactive** - `while` is not live; the lock does nothing and the stage runs normally (the cheap path). **Held** - `while` is live and `until` is not yet published; the stage is kept OUT of `route` (the dispatch list) and reported in a separate `held` map keyed by the unmet `until` signals - a held stage produces no output, so any downstream consumer left without a producer is dropped too. **Released** - `until` goes live; the lock clears and the stage rejoins the route. Multiple locks **AND** together (every one must be inactive or released for the stage to run). `while` and `until` match on family prefix, like every other signal. A lock is a **scheduling gate, not a data input** - it gates *when* a stage may run, never *what* it reads.

- The `system-executor`'s `{while:#destructive-op, until:#safety-approved}` lock is the **safety gate**: on a destructive or irreversible system step, `system-planner` (or `triage`) publishes `#destructive-op`, so the executor is held until the `safety-gate` stage carries the action to the user and `#safety-approved` fires - no destructive command runs unconfirmed.
- The `ship-executor`'s `{while:#ship-ready, until:#ship-approved}` lock is the **ship gate**: the orchestrator emits `#ship-ready` at convergence on a ship request (see `## Shipping`), and the `ship-gate` stage - a real `routes:[code]` stage - carries the forward git/gh commands to the user and publishes `#ship-approved` in-route. No commit, push, or PR runs unconfirmed.
- The `code-implementer`'s `{while:#needs-tests, until:#tests-ready}` lock is the **TDD gate**: on a logic build `triage` publishes `#needs-tests`, so the implementer is held until `test-review` validates the red tests and publishes `#tests-ready` - code cannot start against unvalidated tests. A trivial change carries no `#needs-tests`, so the lock is inactive.
- Both implementers also carry a second `{while:#plan-ready, until:#plan-approved}` lock - the **plan-approval gate**. `#plan-ready` fires the moment a plan exists, so this lock is armed on every planned code or system build; no planned change ever starts against an unapproved plan. The `#direct-impl` short path never publishes `#plan-ready`, so the lock is inactive by silence and the implementer runs straight off `@confirmed-intent`.

**Release policy.** `#plan-approved` has **no in-route publisher** on the system or small planned-build path (`plan-challenger` is `routes:[code]` and only runs on a `#significant-build`), so on those paths the orchestrator emits `#plan-approved` as a **HARD REQUIRED step**. The cue is a stage sitting in `held` keyed on `#plan-approved` with no in-route producer; the `#direct-impl` short path never reaches this policy. (The threshold below is the De Morgan dual of the `## Gates` plan-approval paragraph - `>1 file or est-size > S` - kept in sync by this cross-reference.)
- **System** - always a one-tap confirm before execution (no `plan-challenger` band to fall back on, which is why the emit must be hard and backstopped). The confirm asks "Run this system change?", header `Run change`, options Proceed / Abort.
- **Small planned build** - auto-publish `#plan-approved` iff the plan touches `<=1` file AND `est-size <= S`; otherwise a one-tap confirm asking "Proceed with this change?", header `Proceed`, options Proceed / Hold - declining escalates into the full `plan-challenger` band (publish `#significant-build`; the challenger then owns the release). When `#est-size` is not live, treat the plan as NOT small - confirm, never auto-publish. `est-size` stays advisory for stage selection but is **load-bearing** in exactly two decisions: this small-planned-build auto-release and triage's `#direct-impl` publication (`agents/triage.md`) - when absent, the safe default is confirm.

Each one-tap confirm renders the planner's Plan Breakdown in its card, and the gate's concrete example rides in that card breakdown, not in the option descriptions (which stay fixed-process example-exempt). On the `code` path proper (`#significant-build` live), `plan-challenger` publishes `#plan-approved` on Approve, so the orchestrator does not emit it there; on a multi-plan run `plan-arbiter` is the sole in-route publisher (its Adopt verdict), while Hybrid and Revise-first re-spawn the planner and publish nothing (`doctrine/multi-plan.md`).

## Milestone loop

A `#significant-build` decomposes into milestones reviewed one increment at a time. The planner's `## Plan Breakdown` always carries an ordered, **advisory** milestone list (`agents/code-planner.md`); the orchestrator owns the loop/no-loop choice.

- **Arming (orchestrator rule).** The orchestrator ENGAGES the loop - per-milestone execution, the EARLY pass, and the divergence re-split - if and only if `#significant-build` is live; otherwise a single pass, byte-AND-cost-identical to today (the N=1 / cheap path, no early pass). A second, orthogonal choice off that same signal is whether to run **multiple plans in parallel**, each critiqued, then adjudicated by `plan-arbiter` - armed when `#significant-build` is live AND the design space is wide; see `doctrine/multi-plan.md`.
- **EARLY pass (per milestone, N>1 only).** Only the milestone-scoped lenses run over that milestone's diff slice: `correctness-reviewer` + `security-reviewer` (surface-gated) + `shape-reviewer` + `test-verifier` (smoke), each `milestone-scope`-tagged. The full End Review wave fires once at the end (HARD withhold below).
- **Construction invariant.** The implementer publishes `#milestone-diverged` only during loop execution; loop execution requires `#significant-build`; therefore `plan-challenger` (which subscribes it) is always live when a divergence re-split composes - the dead-subscriber state is unreachable by construction.
- **Milestone-boundary spec.** At each boundary the orchestrator (a) live-set removes `#tests-ready` / `#tests-red` / `#test-cases-ready`; (b) KEEPS `#needs-tests` / `#significant-build` live so the implementer's locks go HELD (not inactive); (c) drops `{test-plan, test-author, test-review}` from `already_run` so the TDD chain re-runs - without (c) the re-held implementer would stall with no in-route producer for its `until` signal. Symmetric to the `system-verifier` drop-from-`already_run` (`## Convergence`).
- **Forward-only re-split.** `#milestone-diverged` re-splits the REMAINING milestones k+1..N only; the shipped milestones 1..k are never re-gated (mirrors "Late escalation, never a re-gate", `## Convergence`).
- **HARD withhold (HARD REQUIRED, same emphasis as the `#plan-approved` orchestrator-emit).** Until `#milestones-complete` (orchestrator-emitted), the orchestrator MUST NOT add `@diff` to `available` and MUST NOT publish `#code-written` - the End Review wave fires exactly once, after the final milestone. No git commits between milestones; the working tree accumulates and only the final cumulative diff is the artifact the End wave sees (consistent with `## Recovery`'s reconcile-from-tree). For an XL cumulative diff, native `/code-review` (ultra thinking) is the documented alternative.
- **Tier-growth re-gate.** An orchestrator observation, not a signal: when a re-split grows the remaining work past its `est-size` tier, the orchestrator retracts `#plan-approved` via live-set removal (`## Convergence`, Stale-approval retraction) and re-gates - one-tap - the REMAINING milestones only. The retraction lands BEFORE the next milestone's implementer run, so it stays pre-implementation for that milestone. During the loop the card shows the milestone-status layer atop the per-stage markers (`doctrine/render-card.md`).

## Shipping

The shipping tail is **opt-in and in-session only**: it ships the work built THIS run as one commit, pushed directly to the base branch (default) or on a feature branch with a draft PR - the user picks the target at the ship gate, which carries the commands and recovery for each. It engages only when the request explicitly asks to ship, release, or open a PR - `triage` then publishes `#ship-requested` (a marker alongside `code`, never its own path). With no such request the tail never composes and the run converges normally. The ship gate is **surfaced only at convergence** - a deliberate appendix after the build is done. The trigger is a HARD REQUIRED orchestrator emit, the same shape as the `#plan-approved` emit (`## Locks`, Release policy): at convergence (`## Convergence`), when `#ship-requested` is live AND `#ship-ready` is not, the orchestrator emits `#ship-ready`. That re-populates the route (`ship-gate`) and `held` (`ship-executor`, `## Locks`); either ship choice publishes `#ship-approved` in-route, releasing the executor, and true final convergence follows the executor's `#shipped`. `#ship-ready` sits in `SEED_SIGNALS` (`hooks/check_catalog.py`) because it is orchestrator-emitted, not stage-published - the same basis as `#request-received` and `#critiques-ready`; the generic seed cost applies, since the seed exempts the topic from the orphan publisher check.

## Input Template Contract

Every agent receives inputs via a tagged-slot template authored under the agent's `## Input` heading and compiled into `generated/catalog.json` as `input_template`, so the orchestrator fills slots from catalog state without opening each agent's `.md`. The main agent fills slots verbatim from predecessor output - no paraphrase. Every template names each required slot with an XML-style tag (e.g. `<CONFIRMED_INTENT>`, `<SCOUT>`, `<APPROVED_PLAN>`) and states the source agent and expected content; the agent's first step parses required slots and on a missing one emits `INPUT_ERROR: missing <slot>` and stops. Output wrapping mirrors it: agents emit structured blocks named with XML-style tags that successors reference (e.g. `<APPROVED_PLAN version="N">`, `<CLARIFY_OUTPUT>`), making relay mechanical and enabling re-injection after compaction.

### Artifact handles

- **artifacts on disk, handles in context.** When the same large `<APPROVED_PLAN>` must reach many consumers across a run, the orchestrator writes the plan to disk ONCE and hands each consumer a short handle - a read-imperative envelope: `Read the verbatim <APPROVED_PLAN> at <path> - its bytes ARE the artifact.` The bytes on disk ARE the verbatim artifact; the handle just defers the read.
- **Handle consumers.** The five parallel End Review reviewers - `acceptance-reviewer`, `conventions-reviewer`, `correctness-reviewer`, `shape-reviewer`, `simplicity-reviewer` - PLUS `code-implementer`, `plan-challenger`, `fixer`, `test-plan`, `test-review`. Each carries the in-body handle-read line (after its `## Input` block), which activates only when the slot actually holds a handle, so it never misfires on an inlined consumer. Two inline carve-outs: `plan-arbiter` (a multi-plan run reads N distinct competing plans, which one shared `plan-<slug>.md` cannot serve - the same reason `plan-challenger` takes the handle ONLY on a single-plan run) and `safety-gate` (solo gate consumer on the system route; the indirection buys nothing).
- **Union gate (state it exactly).** Offload only when the plan block exceeds `RIVER_ARTIFACT_OFFLOAD_CHARS` (a fixed documented threshold, ~1500 chars - a constant of this doctrine, not a runtime-read knob) AND ( the parallel End Review wave carries more than one plan-consuming reviewer OR `#significant-build` is live ). `correctness-reviewer` fires on every code build and `simplicity-reviewer` on every planned build, so even a non-significant planned build with a >1500-char plan forms a >1 plan-consuming wave and must still offload; the `#significant-build` term adds the sequential consumers. A smaller plan stays inlined - both conditions must hold.
- **Write timing.** The orchestrator DECIDES the offload and the `code-planner` PERFORMS the write. When the union gate fires the orchestrator computes `<slug>` - a short kebab-case reduction of the plan's subject (lowercase, non-alphanumerics to `-`, collapse `-` runs, trim, cap 40 chars) - and hands the planner the target path `.alp-river/artifacts/plan-<slug>.md` in `<ARTIFACT_OFFLOAD>`; the planner writes the verbatim plan block there in the same spawn that produces it. The size threshold has ONE owner: the orchestrator owns the single `RIVER_ARTIFACT_OFFLOAD_CHARS` measurement and re-applies it to the returned block for the handle-vs-inline decision - the producer never re-measures. A plan that satisfies the union condition but comes back under threshold is still written yet inlined - the price of single ownership. On a Revise or forward-only re-split the same path is supplied, so the planner overwrites the file in place - at a milestone BOUNDARY only (before the next milestone's implementer run, NEVER in-flight, which would swap the artifact under a consumer still reading it); absent a re-split nothing rewrites. The file is refreshed, not appended - the latest write is the one every consumer reads.
- **input_template is UNCHANGED for the handle-vs-inline mechanism.** The handle rides at fill-time only: every consumer's template still declares the `<APPROVED_PLAN>` slot, and the orchestrator chooses, per the union gate, whether to fill it with the inline block or the handle envelope. The one genuine `input_template` addition is `code-planner`'s own `<ARTIFACT_OFFLOAD>` producer slot (Write timing, above).

## Revision Contract

This harness has no live follow-up to a running subagent: every re-production of a prior output is a fresh spawn whose input must be self-contained - the orchestrator assembles the package, not the agent (Context Discipline) - deterministic and compaction-survivable, which a live handle is not. Re-spawn fresh and present it as the normal move; after compaction, a revision in flight is reconstructed from the canonical run state plus the prior artifact (`## Compaction`). A revision package fills three roles, each via the agent's own speaking-named slots (no shared generic tag):

1. **Prior version** - the complete prior artifact: folded into the prompt verbatim when it lives only in the conversation (the plan), or re-read from disk when it is a file the agent wrote (the tests).
2. **Corrections** - the exact directive driving the change (challenger `BLOCKERS`, implementer kickback `REASON`, test-review misalignment report).
3. **Guard** - the literal instruction to reproduce the prior version exactly except where a correction applies, emit a minimal diff rather than a from-scratch re-derivation, and bump the version where versioned.

- **Correction revision** (guard applies; re-emits the agent's OWN artifact, stable tag, bumped version): `code-planner` - prior version `<PRIOR_PLAN>`, corrections `<REPLAN_REASON>`, emits `<APPROVED_PLAN version="N+1">`; `test-author` - prior version is the on-disk tests it wrote (re-read, not folded in), corrections `<TEST_CORRECTIONS>`, amends the suite in place.
- **Non-revision boundary.** An agent making forward edits to a shared mutable artifact from OTHERS' findings is not revising - `fixer` edits the live working tree from reviewers' findings each round, and its `<ROUND>` is an oscillation counter, not a prior-version fold. The guard does not apply.
- **Convergence loop.** Re-derives an evolving artifact as new user input arrives; carries `<PRIOR_ROUNDS>` and folds in the new answers, with NO verbatim guard - the artifact is meant to change round over round. Instances: `clarifier` (convergence-governed), `setup-agent` (invocation-capped). See `## Clarification Loops`.
- **Not a revision (phases).** `design-prototyper` (`confirm-params` then `built`), `ux-prototyper` (`confirm-flow-params` then `built`), `capture-agent` (`PROPOSAL` then `WRITE`) - sequential phases each emit a different artifact and carry forward the user's picks. No guard.

## Compaction

`hooks/recover-run-state.sh` is the single SessionStart recovery hook, registered on startup, resume, and compact. It re-anchors the workflow pointer plus the canonical run state (the current route, live signals, available artifacts, and premises), reading **file-first** from the per-run `.alp-river/runs/<run-id>/run-state.json` (`## Durability`). When that file is absent or invalid it emits the workflow anchor plus a preserve-manually note - the on-disk file is the sole state source; there is no transcript recovery. The router recomputes the route from the re-anchored blocks, so resumption is deterministic. Preserve manually only what is not in those blocks: the stage currently mid-run and any gate awaiting the user - a pending plan-approval confirm waiting on the orchestrator-emitted `#plan-approved` is one such gate, preserved by that same rule.

## Code Quality
- Use the project's formatter.
- Failing tests → fix the code, keeping assertions and coverage intact.
- Delete dead code: unused functions, stale imports, obsolete files.
- Search for existing patterns before writing new ones. Reuse beats reinvention.
- Improve the area you're touching: dead code, stale abstractions, obvious simplifications.

## Changelog Style

The CHANGELOG.md entries and the README's "Latest updates" section follow the same rules: a user-facing summary, not implementation notes. README's "Latest updates" mirrors the CHANGELOG entry verbatim - same wording, same bullet count.

- **Shape by release type:** patch (X.Y.Z, Z > 0) - bullets only, no intro; minor (X.Y.0) - one intro paragraph framing the release theme, then bullets; major or initial release (X.0.0, or 0.1.0) - several intro paragraphs giving the wider arc, then bullets.
- **Per-bullet rules:** one line per bullet - no multi-sentence run-ons; outcome only - what changes for the person using this, not how it works inside; no internal terms (step numbers, agent names, axis names, flag names - if a stranger wouldn't recognize the word, drop it); no clever framing (no rhetorical asides, no "but"/"also"/"instead" - state the change and stop); bullets cap at four per release.
